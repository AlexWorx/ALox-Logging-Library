A - Namespaces

The implementation parts of the ALox library are split into different namespaces (in Java called "packages").

All A-Worx software namespaces start with 

    com.aworx       // Java
    cs.aworx        // C#
    aworx           // C++

This is the rational behind this setup:

  1. The Java language standards just demands things to be that way.
  2. The other language standards were never so explicit like the Java was (and were never threatening with being arrested and even worse things if one does not apply!)
  3. We are in need to have different prefixes, otherwise our preferred documentation system (we use the unrivaled coolest tool Doxygen!) gets irritated due to ambiguous symbols.

A design principle of namespaces in ALox is that the relationship between namespaces is not bidirectional. This means that if one class in a namespace A inherits or uses a class in a namespace B, then there is no reference from (any) class namespace B into namespace A. You can say here that "if A knows B then B must not know A".

Lets start with two namespaces:

  • [com.|cs.]aworx.lox (C++, C#, Java)
    This is the main and basic namespace of ALox. The central class is Lox (C++, C#, Java), which acts as a container for the Loggers, manages a set of different settings and provides the user interface (API) into ALox logging.
  • [com.|cs.]aworx.lox.core (C++, C#, Java)
    As the name says, within this namespace (package) you will find the core classes needed for logging. Besides the abstract class Logger (C++, C#, Java), there are classes that care about internal things like scope information or Log Domains.
    In addition, utility classes for certain Logger types are located here (currently for TextLoggers, probably more to come in the future). A normal user of the ALox Logging Library would not use classes of this namespace directly.

Now, namespace aworx.lox knows namespace aworx.lox.core, but not vice versa:

A third namespace is:

  • [com.|cs.]aworx.lox.loggers (C++, C#, Java)
    This namespace provide ready to use Logger types, e.g. AnsiConsoleLogger (C++, C#, Java), or MemoryLogger (C++, C#, Java).

Neither aworx.lox.core nor aworx.lox know about this namespace. Of-course, aworx.lox.loggers knows about aworx.lox.core. This makes sense, because instances of Lox get instances of abstract type Logger attached. All that class Lox knows is that they are derived from class Logger in namespace aworx.lox.core. This allows you to create your own loggers.

Besides understanding a little about the namespaces strategy of ALox, a simple takeaway from this is that you need to make namespaces aworx.lox and aworx.lox.loggers available to your code, but not aworx.lox.core.

Note
As C++ supports it nicely, in this implementation we are using a special mechanism to get an 'alias' of all major classes into namespace aworx. This is explained technically in C++ namespace aworx and means that a simple using aworx; statement is enough to 'see' ALox as well as ALib classes.

Finally, there is a "underlying" namespace called aworx.lib implementing ALib. This is pure, basic utility code. Some of the classes are just faster versions (more Bauhaus!) than equivalent existing system classes. Others do exist only to enable ALox code to look the same across languages and platforms. Of-course, you can read the reference documentation and you might use them somewhere, but this is off-topic here. There are many other libraries existing that are much more complete and standard. Again, we are building upon them, only because we really think it makes our life easier in respect to porting ALox and other of our projects and because it makes your logging life just faster.

As every rule has an exclamation, so this one has. For example, we broke the "if A knows B" then "B does not know A" rule intentionally by adding basic 'structs', constants and enums, like Verbosity (C++, C#, Java) or Scope (C++, C#, Java), to to the outer, main namespace [com.|cs.]aworx.lox (C++, C#, Java) for convenience. The same applies to class ESC (C++, C#, Java). And when you have a closer look into the source code you might find other minor samples.
