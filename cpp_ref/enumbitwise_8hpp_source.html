25 /**
26  * # Introduction #
27  *
28  * With scoped enums, introduced by C++ 11, logical operators are not supported. This seems a
29  * little in contradiction as still enums can be defined bitwise. What we mean with the term
30  * "bitwise" is that the elements of enums get numbers assigned which each have set a different
31  * bit in the underlying integer type.
32  *
33  * As a sample, consider the following two scoped enum types:
34  *
35  * \snippet "DOX_ALIB_LANG.cpp" DOX_ALIB_LANG_ENUMS_NORMAL
36  * \snippet "DOX_ALIB_LANG.cpp" DOX_ALIB_LANG_ENUMS_BITWISE_DECLARATION
37  *
38  * \note The first enum is taken from the sample found in documentation of TMP struct
39  * \alib{lang,T_EnumMetaDataDecl}. We have applied macro \ref ALIB_LANG_ENUM_NAMED also to the
40  * second, bitwise enum type. For further
41  * information refer to \alib{lang,T_EnumMetaDataDecl,ALib Enum Meta Information} and below.
42  *
43  * The second enum type obviously is bitwise. And as the identifier indicates that values
44  * of this enum represent "states of windows of a window manager", it is obvious that
45  * enum values with multiple bits set might occur.
46  * Nevertheless, the following code
47  *
48  * \snippet "DOX_ALIB_LANG.cpp" DOX_ALIB_LANG_ENUMS_BITWISE_SAMPLEERROR
49  *
50  * leads to a compilation error similar to this:
51  *
52  * error: invalid operands to binary expression ('WindowManager::States' and 'WindowManager::States')
53  *
54  * Therefore, if operators are needed, the common approaches are to either
55  * - define operators for each scoped enum type that need them, or to
56  * - provide a set of templated operator functions that handle all scoped enum types.
57  *
58  * The first approach includes a lot of duplicate code, the second has the disadvantage that it
59  * undermines the intention of the C++ 11 language standard: The compiler would allow the operators
60  * even non-bitfield custom scoped enum types.
61  *
62  * # %ALib Bitwise %Enum Operators #
63  *
64  * This extremely simple TMP struct is used to overcome the disadvantages of the approaches discussed
65  * above. The default implementation is empty and just inherits from \c std::false_type.
66  *
67  * Specializations of the struct for a certain enum type should be performed using macro
68  * \ref ALIB_LANG_ENUM_IS_BITWISE. The macro needs to be located in a header file, probably
69  * close to the enum type declaration itself, but \b outside of any namespace (aka in the default
70  * namespace). In our sample, we simply put the code line:
71  *
72  * \snippet "DOX_ALIB_LANG.cpp" DOX_ALIB_LANG_ENUMS_BITWISE_SPECIALIZE
73  *
74  * at the end of the header file (again: outside of the namespace!). All that the macro does, is
75  * specializing this struct to inherit from \c std::true_type. (With the current version of \alib,
76  * still the struct is empty.)
77  *
78  * With this specialization, the compiler accepts a set of templated operator functions which
79  * get conditionally selected using TMP struct \c std::enable_if which of-course checks for
80  * this struct to be specialized for the type in question.
81  *
82  * The following code snippet now compiles:
83  *
84  * \snippet "DOX_ALIB_LANG.cpp" DOX_ALIB_LANG_ENUMS_OPERATORS_SAMPLE_1
85  *
86  * The following \alib set of operators are applicable to scoped enumerations of bitwise type:
87  *
88  * - \b operator& (Link to documentation of ampersand operator currently impossible for technical reasons.)
89  * - \b operator&=
90  * - \ref operator|
91  * - \ref operator|=
92  * - \ref operator^
93  * - \ref operator^=
94  * - \ref operator~
95  * - \ref operator+ (An alias for \ref operator|=)
96  * - \ref operator- (Removes bits of rhs in lhs)
97  * - \ref operator+= (An alias for \ref operator|=)
98  * - \ref operator-= (Removes given bit(s) )
99  *
100  * In addition, helper function
101  * - \ref EnumContains
102  *
103  * is available.
104  *
105  * Independent from this TMP struct, for all C++ scoped enum type, in addition, helper function
106  * - \ref EnumValue
107  *
108  * is provided with including this header file.
109  *
110  * \attention
111  * While in this documentation the operators are appearing under namespace <c>aworx::lib::lang</c> ,
112  * in reality they are defined in <b>no namespace</b> (global namespace).
113  * This was done to have the documentation of the operator functions and this struct just in one place!
114  * \attention
115  * Due to the template meta programming, this does not â€™clutter' the global namespace and does
116  * not interfere with any existing user code, while still there is the advantage of
117  * not needing to put a using statement like
118  *
119  * using namespace aworx::lib::lang;
120  * to each source location that use the operators.
121  *
122  *
123  * \~Comment #################################################################################### \~
124  * # Bitwise Enums and %ALib %Enum Meta Data #
125  *
126  * This concept of supporting bitwise operations on selected types, goes very well along with
127  * the \alib{lang,T_EnumMetaDataDecl,ALib Enum Meta Information} features.
128  * However the use of a "element to enum identifier name translation table" as described and sampled
129  * in this context, needs a little more thinking. Because bitwise enum values might consist of
130  * mixed elements, the simple approach to find just one corresponding meta data table entry
131  * for converting an enum element into a string representation fails. In contrast, probably more
132  * than one element has to be written or combinations of bits set as once, should even lead to
133  * new words, not defined in the enum class itself (e.g. "Maximized" in the sample above).
134  *
135  * Therefore, \alib provides two different specializations of TMP struct \alib{strings,T_Apply},
136  * namely
137  * - \alib{strings::applyto,T_Apply<TEnum>}
138  * for applying non-bitwise enum types to \b %AStrings.
139  * - \alib{strings::applyto,T_Apply<TEnumBitwise>}
140  * for applying bitwise enum types to \b %AStrings.
141  *
142  * Again, they are selected by the compiler automatically dependent on the availability of
143  * a specialization of this struct \b %T_EnumIsBitwise, using \c std::enable_if.
144  * That is good news, of-course, but when defining the "translation table", some special
145  * rules (and options!) have to_be kept in mind. Those are discussed and sampled with
146  * documentation of
147  * \ref aworx::lib::strings::applyto::T_Apply<TEnumBitwise> "T_Apply<TEnumBitwise>".
148  *
149  *
150  * # Reference Documentation #
151  *
152  * @tparam TEnum The enum type to provide information for.
153  * @tparam TCheckEnum This parameter has a default expressions and <b>must not</b> be provided
154  * with specializations of this struct.
155  * It is used internally to ensure that only enum types are to be used with
156  * the first template parameter \p TEnum.
157  */
158 template<typename TEnum,
159  typename TCheckEnum= typename std::enable_if<std::is_enum<TEnum>::value>::type>
160 struct T_EnumIsBitwise : public std::false_type {};

192  *
193  * Selected by the compiler only if \alib{lang,T_EnumIsBitwise} is specialized for
194  * template enum type \p TEnum to inherit \c std::true_type.
195  *
196  * @param lhs First operand.
197  * @param rhs Second operand.
198  * @tparam TEnum Enumeration type.
199  * @return The result of a bitwise and operation of the underlying enum values.

204 operator& (TEnum lhs, TEnum rhs) noexcept(true)
