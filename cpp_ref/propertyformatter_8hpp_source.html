25 /** ************************************************************************************************
26  * # Introduction # {#alib_namespace_strings_propertyformatter_overview}
27  *
28  * This template class can be used to offer customized format strings to end users.
29  * For this, the format string definition of \b %ALib
30  * \ref aworx::lib::strings::format::Formatter "Formatter" types is extended to support identifiers,
31  * which act as placeholders for object values. The identifiers placed in the string have to match
32  * to the normal formatting placeholders in respect to their total number and value type.
33  *
34  * In the constructor of this class, a custom format string which is based on standard \b %ALib
35  * format strings is passed. This string is processed as follows:
36  * - all custom identifiers are removed
37  * - for each identifier found, a reference to an associated callback function is stored.
38  *
39  * Hence, the translation of the custom format string into \b %ALib formatter strings and the
40  * selection of the callback functions is done only once.
41  *
42  * The only interface method #Format then accepts a target string object and a reference to an
43  * object of custom type, which is is defined by template parameter \p TFormattable.
44  *
45  * Using this template type, a vector of tuples is defined and need to be passed to the constructor
46  * of this class. The first element of each tuple contains the \e 'identifier' string which is
47  * removed from the format string if found. The second element contains a callback function used
48  * to retrieve the actual data.
49  *
50  * The whole approach is useful in cases where an application wants to allow a customizable output
51  * format of data objects.
52  *
53  * # Sample # {#alib_namespace_strings_propertyformatter_sample}
54  * Let us assume, an application uses an enumeration and a simple struct:
55  * \snippet "DOX_ALIB_STRINGS.cpp" DOX_ALIB_STRINGS_PROPERTY_FORMATTER_1
56  *
57  * In the applications' configuration file, a custom output format for objects of type \b Person
58  * should be user defined. We document to the user that he can use Python-style or
59  * Java-style output formats - extended by the ability to place the following three identifiers in
60  * the string:
61  * - \@name
62  * - \@age
63  * - \@hobby.
64  *
65  * A possible format string then would be:
66  *
67  * "{@name} is aged {@age} and his/her hobby is {@hobby}"
68  *
69  * To tell this class how to retrieve the replacement values, we need to define three callback
70  * functions:
71  * \snippet "DOX_ALIB_STRINGS.cpp" DOX_ALIB_STRINGS_PROPERTY_FORMATTER_2
72  * It is a good idea to place the callback functions in an anonymous (sub-) namespace as they
73  * are only referred to once (in the next step).<br>
74  * As this sample is very trivial, the second parameter <c>AString&</c> is not used and named.
75  * Details on the purpose and use of this parameter is addressed
76  * \ref alib_namespace_strings_propertyformatter_callbacks "below".
77  *
78  * As a next step, these functions need to be collected together in a "translation table".
79  * The table primarily holds a string denoting the replacement identifier and a pointer to the
80  * corresponding callback function. For convenience, the table type is provided with <c>using</c>
81  * defintion #TCallbackTable.
82  *
83  * In our sample, the definition of the table looks like this:
84  * \snippet "DOX_ALIB_STRINGS.cpp" DOX_ALIB_STRINGS_PROPERTY_FORMATTER_3
85  *
86  * This is all we need! Of-course, an external declaration of our table \b PersonCallbacks
87  * should be placed somewhere in the projects' header file. With this, anywhere in the application,
88  * a code like this may now use our custom formatter strings:
89  * \snippet "DOX_ALIB_STRINGS.cpp" DOX_ALIB_STRINGS_PROPERTY_FORMATTER_4
90  *
91  * Running the code above produces the following output:
92  * \snippet "DOX_ALIB_STRINGS_PROPERTY_FORMATTER.txt" OUTPUT
93  *
94  * Serializing objects in a custom format (e.g. a user could define its own \b JSon object output)
95  * should be the most obvious and frequent use case. And this is what the classes name suggests.
96  * Nevertheless, the callback methods might be more complex than just returning "properties" of
97  * objects. The next sections gives more details on the custom callbacks.
98  *
99  * \~Comment * * * * * * * * * * * * * * * * * * * * * * * * * * * * * * * * * * * * * * * * * * \~
100  * # Callback Functions # {#alib_namespace_strings_propertyformatter_callbacks}
101  * The call back functions' signature is defined with <c>using</c> definition #TCallback which
102  * evaluates to
103  *
104  * Box (&)(TFormattable&, AString&);
105  *
106  * The return type is \ref aworx::lib::boxing::Box "Box", which allows the callback function to
107  * return objects of arbitrary type, respectively all types that received support
108  * with \b %ALib formatters.
109  *
110  * The first input parameter provides the reference to the data object passed to method
111  * #Format and is of custom type \p TFormattable).
112  *
113  * The second parameter is an \b AString object which optionally can be used to assemble
114  * string objects in more complex callback methods. Note, that if this string is not
115  * empty after the callback invocation, the result is copied to a string buffer allocated
116  * in the heap memory. In this case, the box object returned by the callback is ignored
117  * and instead the copy of the string is passed to the formatter. If C++ string constants
118  * (e.g <c>"true"</c> are to be returned by the callbacks, those constants do not need
119  * to be copied to the buffer, as their memory is statically allocated. In other words,
120  * the \b %AString buffer provided in the second parameter is needed to be used
121  * only in cases that a string gets assembled in the callback function!
122  *
123  * \~Comment * * * * * * * * * * * * * * * * * * * * * * * * * * * * * * * * * * * * * * * * * * \~
124  * # Identifiers And The Escape Character '@' # {#alib_namespace_strings_propertyformatter_escape}
125  * The escape character used to find custom identifiers in the format string is defined
126  * with template parameter \p TESCIdentifier and defaults to \b '\@'.
127  * This escape character is searched in the format string. If found, the identifier is
128  * read by consuming alphabetic characters. If a 'closing' second occurrence of \b '@' is found
129  * while the identifier is parsed, then this character
130  * gets eliminated from the custom format string as well. Hence, the two format strings
131  *
132  * "{@name}"
133  * "{@name@}"
134  * are both valid and equal.<br>
135  *
136  * In case of
137  * \ref aworx::lib::strings::format::FormatterPythonStyle "FormatterPythonStyle" format strings,
138  * it is a good idea to place the identifier right inside the brackets. It just looks very intuitive.
139  * However, these versions:
140  *
141  * "@name{}"
142  * "{}@name"
143  * are also valid custom format strings.
144  *
145  * In case of
146  * \ref aworx::lib::strings::format::FormatterJavaStyle "FormatterJavaStyle", which uses \b '\%'
147  * as its escape character, we consider the best option to put the custom identifier in front
148  * of each \b '\%' symbol. The string of the example given above would then look like this:
149  *
150  * "@name%s is aged @age%d and his/her hobby is @hobby%s"
151  * Doing it this way, the \b '\%' symbol acts as a very natural delimiter for the custom identifier.
152  *
153  * As a final note, with the second element of tuple \ref TCallbackTableEntry, it is possible
154  * to allow abbreviations of identifier names. This column denotes the minimum
155  * number of characters to be matched. As in the above sample a value of \c 1 is provided, each
156  * identifier of the custom format string can be abbreviated down to one character. Consequently
157  * the following format string samples are all equal and allowed:
158  *
159  * "{@name}"
160  * "{@nam}"
161  * "{@na}"
162  * "{@n}"
163  *
164  * <p>
165  * <p>
166 \~Comment ####################################################################################### \~
167  * # Reference Documentation #
168  * @throws aworx::lib::strings::format::Exceptions::UnknownPropertyInFormatString
169  *
170  * @tparam TFormattable The type that is used to retrieve the format arguments from
171  * @tparam TESCIdentifier The prefix used to search identifiers in the user defined format
172  * string. Defaults to \e '\@'.
173  **************************************************************************************************/

226  /**
227  * Constructor. Processes the given format string and builds internal structures
228  * to be able to quickly process invocations of method #Format.
229  *
230  * @param customFormatString The format string as described in the class documentation.
231  * @param propertyTable Table with property identifier names and callback functions to
232  * retrieve the property values.
233  * @param formatter The formatter to use. Defaults to \c nullptr which selects
234  * \b %ALib \ref aworx::lib::strings::format::Formatter::AcquireDefault "default formatter".
235  *
236  * @throws aworx::lib::strings::format::Exceptions::UnknownPropertyInFormatString
237  */
238  PropertyFormatter( const String customFormatString,
239  const TCallbackTable& propertyTable,
240  Formatter* formatter= nullptr )
241  {
242  theFormatter= formatter;
243  propertyFormatString=
244  formatString= customFormatString;
245  integer parsePos= 0;
246  while(parsePos < formatString.Length() )
247  {
248  // find next parse position
249  integer parsePosCopy= parsePos;
250  if( (parsePos= formatString.IndexOf( TESCIdentifier, parsePosCopy ) ) < 0 )
251  break;
252 
253  // find end of Identifier
254  integer endPos= parsePos+ 1;
255  while( endPos < formatString.Length() && isalpha( formatString[endPos] ) )
256  endPos++;
257 
258  String identifier( formatString, parsePos + 1, endPos - parsePos - 1 );
259 
260  // found
261  if( !identifier.IsEmpty() )
262  {
263  // add callback
264  auto callBack= lib::strings::util::FindStringStartInTupleVector<Case::Ignore, 0, 1>
265  ( propertyTable, identifier );
266  if( callBack == nullptr )
267  {
268  Exception e( ALIB_SRCPOS_REL_NULLED, Exceptions::UnknownPropertyInFormatString,
269  TESCIdentifier, identifier, customFormatString );
270  Boxes& args= e.Entries.back().Args;
271  for( auto row : propertyTable )
272  {
273  args.Add( TESCIdentifier );
274  args.Add( std::get<0>( row ) );
275  args.Add( ", " );
276  }
277  args.back()= '.'; // replace the last comma
278 
279  throw e;
280  }
281  // remove identifier from format string
282  formatString.Delete( parsePos, endPos - parsePos + (formatString[endPos] == TESCIdentifier ? 1 : 0) );
283 
284  // store callback
285  callBacks.emplace_back( callBack );
286  }
287 
288  parsePos++;
289  }
290 
291  }

293  /**
294  * Writes the formatted output of the properties of the given \p TFormattable object
295  * to a given target string.
296  *
297  * @param target The target string to write into.
298  * @param src The custom object which is passed to the callback methods to collect
299  * the formatter arguments.
300  */
301  void Format( AString& target, TFormattable& src )
302  {
303  // string buffers
304  std::vector<String> heapStrings;
305  String128 localString;
306  ALIB_WARN_ONCE_PER_INSTANCE_DISABLE( localString, ReplaceExternalBuffer );
307 
308  // collect boxes
309  results.reserve( callBacks.size() + 1 );
310  results.Add( formatString );
311  for( auto entry : callBacks )
312  {
313  Box argument= std::get<2>(*entry)( src, localString );
314  if( localString.IsNotEmpty() )
315  {
316  heapStrings.emplace_back( localString );
317  argument= heapStrings.back();
318  }
319 
320  results.Add ( argument );
321  localString.Clear();
322  }
323 
324  // invoke ALib default formatter
325  Formatter& formatter = theFormatter == nullptr ? Formatter::AcquireDefault( ALIB_SRCPOS_REL_EMPTY )
326  : *theFormatter;
327  try
328  {
329  formatter.Format( target, results );
330  }
331  catch(Exception& e)
332  {
333  if( theFormatter == nullptr )
334  Formatter::ReleaseDefault();
335  e.Add( ALIB_SRCPOS_REL_NULLED, format::Exceptions::ErrorInResultingFormatString,
336  propertyFormatString );
337  throw;
338  }
339  if( theFormatter == nullptr )
340  Formatter::ReleaseDefault();
341 
342  results.clear();
343  }
