aworx::lib Namespace Reference

Namespace Description

Introduction:
This is the C++ base namespace for ALib - a set of core utility classes developed by A-Worx GmbH, Germany, published under Boost Software License.

Besides C++, this library is available in C# and Java, with language specific differences. The design goals are:

  • Be small, efficient and easy to understand
  • Keep C++, C# and Java versions similar and mimic functionality of core classes of all three languages whenever this is appropriate, with the intent to keep source code building upon the library to a certain degree similar and exchangeable.
  • Be non-intrusive in the sense that the use of ALib in projects (maybe just because some minor functionality is wanted) does not force to show up many ALib types.

Organization of Code:
The C++ version is the most complex language version of ALib. The reason is that Java and C# languages are equipped with many built-in classes that do not need a replacement. This comfortable situation is different with C++. Therefore, "ALib for C++" has a much more comprehensive mandate and it is more likely that ALib for C++ is used in single language (hence C++ only) projects. In other words, the Java and C# versions of ALib are especially useful, if multi-language software is written (e.g. a phone device app, available as a set of different, native incarnations) to reach a certain degree of code compatibility.

Currently, the full version of ALib is not available as a "stand-alone library". With GitHub project

  • ALox Logging Library,

ALib is distributed in a bundle with ALox, a library for debug- and release logging, again available in C++, C# and Java.

However, on the other hand, subsets of ALib are available with so called ALib Modules. Modules can be used separately from the rest of the library and for a users convenience they are exposed in separated GitHub projects.

The following portions of ALib are extracted and maintained as modules:

  • ALib for C++ Singleton
  • ALib for C++ Boxing
  • ALib for C++ Strings
  • ALib for C++ Boxing & Strings
  • ALib for C++ Configuration

The rational for this deployment strategy (to have either a subset of ALib or a bundle with ALox ) is that we anticipate users to either want to use just a specific ALib innovation, or just all of it. In the latter case, ALox is a perfect extension to what ALib offers.

Note
In the rare case that full ALib is wanted without ALox, then it is quite simple to extract full ALib from the ALox project, because
  • The sources are separated in different folders.
  • There are no dependencies from ALib sources to ALox.
  • Two separated CMake build files for ALib and ALox exists.

Usage
Regardless of using full ALib or using an ALib Module, header file alib/alib.hpp is to be to be included prior to including necessary specific other headers. The distribution-specific header file alib/distribution.hpp will select the right (available) module header files. In the case of modules, source files have to be 'manually' added to the build process (either to a library or directly to a main executable) as demonstrated in the samples deployed with the modules.

Furthermore, some initialization method specific to the ALib Module in question might be needed to be invoked during bootstrapping. With the complete distribution of ALib, such initialization is performed in aggregated method ALib::Init.

Note
When - as recommended - the full version of ALib is used bundled with ALox, then the correct header file and initialization method are alox/alox.hpp and ALox::Init.

Platforms/Toolchains:
As of today, ALib for C++ is compiled and tested under the following platforms and compiler toolchains:

  • GNU/Linux, GNU C++ 11, 32-Bit / 64-Bit
  • GNU/Linux, Clang C++ 11, 32-Bit / 64-Bit
  • Windows 10, Microsoft Compiler (MSVC) C++11, 32-Bit / 64-Bit
  • macOS 10, 64-Bit

Adoptions to other platforms should be implementable with limited efforts. All relevant code which selects platform/toolchain specific libraries, intrinsics, etc., will expose a preprocessor error if section fails due to unknown environment. This allows to quickly insert the right platform/toolchain specific code at these places.
