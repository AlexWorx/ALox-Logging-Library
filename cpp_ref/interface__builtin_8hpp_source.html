66 /** ************************************************************************************************
67  * This is one of the built-in \alib{boxing,Interface} classes of <b>%ALib %Boxing</b>.
68  * This interface is invokable with all boxable types, as a default implementation is registered
69  * (see \alib{boxing,Boxer::DefineDefaultInterface}.
70  * The default implementation is provided with this base interface class itself.
71  *
72  * The default implementation compares the type and if it is the same, just compares the stored
73  * raw values against each other. Specifics for array types are \b not implemented.
74  *
75  * This leads to a nested sort order, with the type information being the outer order and
76  * the boxed data being the inner.
77  *
78  * Type-specific implementations should use the following implementation scheme:
79  *
80  * if( rhs.IsType<AComparableType1>() )
81  * return lhs.Unbox<MyTypbe>() < comp.Unbox<AComparableType>();
82  *
83  * if( rhs.IsType<AComparableType2>() )
84  * return lhs.Unbox<MyType>() < comp.Unbox<AComparableType2>();
85  *
86  * if( ...
87  * return ...
88  *
89  * return std::type_index( lhs.GetTypeInfo() ) < std::type_index( rhs.GetTypeInfo() );
90  *
91  * The important thing is the last line: if types are not comparable return
92  * the result of the comparison of the \c std::type_index of the two boxed types.
93  * Only with this, for example \c std::sort will work properly on containers of boxes of
94  * mixed types.
95  *
96  * \note
97  * It is a matter of the compiler (and can not be determined by the user code) how the
98  * types are sorted. Furthermore, the data value might not be reasonably comparable
99  * between boxes of the same type. Reasonably in respect to retrieving a reasonable
100  * sort order.
101  *
102  * Consequently, the use of this comparison operator with custom types is subject to
103  * "knowing what I am doing" and knowing how the comparison is implemented for
104  * those types.
105  *
106  * With boxing \alib{boxing,ftypes,C++ fundamental types}, interfaces are provided that
107  * allow to compare singed against unsigned integers and both against double values.
108  *
109  * This interface is invoked by \alib{boxing,Box::operator<}. The following sample demonstrates
110  * this:
111  * \snippet "DOX_ALIB_BOXING.cpp" DOX_ALIB_BOXING_COMPARE
112  *
113  * It generates the following output:
114  *
115  * \snippet "DOX_ALIB_BOXING_COMPARE.txt" OUTPUT
116  *
117  **************************************************************************************************/

152 /** ************************************************************************************************
153  * This is one of the built-in \ref aworx::lib::boxing::Interface "Interface" classes of
154  * <b>%ALib %Boxing</b>. This interface is invokable with all boxable types, as a
155  * default implementation is registered (see
156  * \ref aworx::lib::boxing::Boxer::DefineDefaultInterface "Boxer::DefineDefaultInterface").
157  *
158  * The availability of this interface is the basis to implement the concept of
159  * <em>"nullable types"</em>, e.g. pointer types or custom types that (already) support the concept
160  * of being \e nullable.
161  *
162  * This default implementation returns \c true if the raw value of box data equals \c 0, otherwise
163  * \c false. This gives a good default behaviour for all boxable types
164  * stored as pointers to the original object.
165  *
166  * Derived class \ref aworx::lib::boxing::IIsNull_false "IIsNull_false" is provided which returns
167  * constant value \c false. This may be registered with custom types that are supposed to never be
168  * \e nulled.
169  * The default implementation of
170  * \ref alib_namespace_boxing_types_ftypes "boxing of fundamental types" will add this
171  * specialization to integer and floating point types. Hence boxed values of these types will
172  * never be considered \e nulled, even if they represent value \c 0.
173  **************************************************************************************************/

249  * to the concept of <em>"nullable values"</em> for non-array types.
250  * In combination with the fact that the default implementation of
251  * \ref alib_namespace_boxing_types_ftypes "boxing of fundamental types" will add a specialization
252  * of \b %IIsNull that returns constant false to integer and floating point types, such types
253  * will never be considered "empty", even if they represent value \c 0.
