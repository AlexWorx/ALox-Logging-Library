48 /**
49  * This type specifies platform independent integer values of the 'natural' bit-size of the
50  * underlying platform. In general, on 32-bit systems this will be 32-bit wide, on 64-bit systems,
51  * 64-bits. Hence, on standard architectures, it has the same bit-size and signedness as
52  * \b std::ptrdiff_t.
53  *
54  * To declare constants of this type, a user defined literal is provided with
55  * \ref aworx::literals::operator "" _integer(unsigned long long int idx) "_integer".
56  *
57  * The type can be considered as a signed version of \c std::size_t. It is needed because
58  * standard type \c 'int' is not specified in respect to its size. E.g. GNU C++ and Clang compiler
59  * use 32-Bit integers for type \c int, even on 64-Bit platforms.
60  *
61  * See also \ref aworx::lib::lang::uinteger "aworx::uinteger".
62  *
63  * \note This documentation is generated using the 64-Bit version of the library. In fact, the
64  * definition as \c int64_t is not guaranteed platform specific.
65  */

76 /**
77  * This type, together with its counterpart
78  * \ref aworx::lib::lang::uintGap_t "aworx::uintGap_t" is used to fill a gap that occurs
79  * when method overloads or template specialization is needed for integer types.
80  * The rational behind and use of this pair of types is best explained with a sample.
81  *
82  * Consider the following code:
83  *
84  * \snippet "DOX_ALIB_LANG.cpp" DOX_ALIB_LANG_INTXX_DECLARATION
85  *
86  * When this is run under 64 Bit - Linux, GNU compiler, the following output is produced:
87  * \snippet "DOX_ALIB_LANG_INTXX.txt" OUTPUT
88  *
89  * This is not what many C++ programmers would expect: Although type <c>long long</c> is the same
90  * 64-bit type as <c>long</c>, the template method is not seen as specialized by the compiler.
91  * Therefore, we have a "gap" in the definition of specializations for types
92  * <c>long long</c> and <c>unsigned long long</c>.
93  *
94  * When compiling and running the same sample code under GNU compiler 32-bit or under
95  * MSVC (Microsoft compiler), 32 or 64-bit, then the gap "moves" to be with types
96  * <c>long</c> and <c>unsigned long</c> instead.
97  * Here, this hurts a lot, because code that uses a simple integer constant \c 1L is not fetched by
98  * the template specializations!
99  *
100  * The lesson learned is that two more specializations are needed and that their types are
101  * dependent on the compiler and library used. Because it is not allowed to specialize
102  * simply with all possible extra variants (this would lead to doubly defined methods),
103  * a preprocessor switch that chooses the right types to fill the gap is needed.
104  *
105  * This type, together with #uintGap_t, does exactly this: using the preprocessor to select
106  * the right "missing" type.
107  *
108  * To fix the sample above, the following two specializations of the template method need to
109  * be added:
110  *
111  * \snippet "DOX_ALIB_LANG.cpp" DOX_ALIB_LANG_INTXX_DECLARATION2
112  *
113  * When overloading functions with integer types, similar rules apply: To have the complete set
114  * of integer types covered, 10 overloads are needed: from type \b int8_t to type \b int64_t,
115  * type \b %intGap_t and then those 5 types in 2 versions, signed and unsigned.
116  * Only with all overloads in place, compiler warnings (on high warning levels) and/or the
117  * necessity of explicit type conversions are avoided.
118  */
119 using intGap_t= platform_specific;
120

124  */
