26 /**
27  * \note
28  * It is important to have a basic understanding of the concept of
29  * \alib{lang,T_EnumMetaDataDecl,ALib Enum Meta Data} to understand the use case of this type.
30  *
31  \~Comment ####################################################################################### \~
32  * ## Introduction ##
33  *
34  * This TMP struct is used to define a certain fixed type of
35  * \alib{lang,T_EnumMetaDataDecl,enum meta data} and to enable a set of macros that easily
36  * assign this type to a custom enum type and to implement a corresponding resource loader.
37  * It could be said, that this struct provides meta information for enum meta data.
38  *
39  * The use case for this concept is given, when a library wants to expose information about the
40  * meta data of enum types it expects with to interface with.
41  *
42  \~Comment ####################################################################################### \~
43  * ## Step By Step Sample##
44  *
45  * The concept provided with this TMP struct and according preprocessor macros is best understood
46  * by looking at a sample.
47  *
48  * \alib itself uses this concept with class \alib{lang,Exception} (among others).
49  * This class is not not designed to be used as a base class for other
50  * exceptions (like for example <c>std::exception</c> is). Instead, it accepts elements of
51  * arbitrary enum types in its constructor. From the enum meta data associated with the
52  * enum element, a first \alib{lang,Exception::Entry,exception entry} is created.
53  *
54  * Then, while the stack is unwinded, catch blocks are allowed to add further entries
55  * and then rethrow the object. This way, each entry is created from an element of a custom enum
56  * type and those types may even be different in each entry!
57  *
58  * The only restriction that the "usable enum types" need to fullfill is that a specific type
59  * of meta data needs to be associated to them. Such meta data includes for example a format
60  * string which is used when the exception is \alox{LogTools,Exception,logged}.
61  *
62  * It is easily understood that \alib exceptions meet exactly the use case of the concept
63  * implemented with this TMP struct and the according macros.
64  *
65  * There are two procedures that need to be followed to use this concept. One for the library
66  * implementation, another for a user of the library that wants to enable custom enum types to
67  * become usable with the library.
68  *
69  * <br>
70  * ### What A %Library Needs To Do ###
71  *
72  * 1. Provide types (usually empty structs named in capital case letters) that denotes the
73  * specifications of the enum meta data.
74  *
75  * In our sample, only one type of meta data is needed. Therefore, the expression class itself
76  * is used. We simply do a forward declaration at the start of the exception's header file:
77  * \snippet "lang/exception.hpp" DOX_ALIB_ENUM_META_DATA_SPECIFCATION_TheType
78  *
79  * 2. Use macro \ref ALIB_ENUM_SPECIFICATION_DECL in a header file to speciialize this TMP struct
80  * (\b %T_EnumMetaDataSpecification).
81  *
82  * Class \b %Exception expects two strings, one holding the enum element name and a second
83  * holding a the resource name of the format string used with method \alib{lang,Exception::Describe}.
84  *
85  * \snippet "lang/exception.hpp" DOX_ALIB_ENUM_META_DATA_SPECIFCATION_decl
86  *
87  * 3. Use macro \ref ALIB_ENUM_SPECIFICATION_IMPL in a compilation unit to implement the
88  * tuple loader once. Note that the heder file for the generic, templated resource loader
89  * object has to be included in the compilation unit.
90  *
91  * In code file \e exception.hpp we find:
92  * \snippet "lang/exception.cpp" DOX_ALIB_ENUM_META_DATA_SPECIFCATION_impl
93  *
94  * 4. Optionally provide a shortcut macro to \ref ALIB_ENUM_SPECIFICATION that predefineds the
95  * custom type of step 1.
96  *
97  * This is done again in the header file of class Exception:
98  * \snippet "lang/exception.hpp" DOX_ALIB_ENUM_META_DATA_SPECIFCATION_macro
99  * <p>
100  *
101  * With these four simple steps, class Exception is ready to accept custom enum types and use
102  * their associated enum meta data. Methods and functions of a library that accept enum
103  * elements of arbitrary type should check whether the type provided uses the right meta data.
104  * This can be done using <c>std::enable_if</c>. To stay with our sample, let us look at
105  * method \alib{lang,Exception::AddFromBoxes} which is called by all other methods that accept
106  * enum types. Hence, it is sufficient to "block" this function for the compiler, if a
107  * wrong enum type is passed:
108  *
109  * \snippet "lang/exception.hpp" DOX_ALIB_ENUM_META_DATA_SPECIFCATION_usinginlib
110  *
111  * <br>
112  * ### What The User Of A %Library Needs To Do:###
113  * With this setup of of a library type, now users of the type can declare their enum types to
114  * be compatible with the library.
115  *
116  * The only thing that the using code needs do for preparation is to use macro
117  * \ref ALIB_ENUM_SPECIFICATION (or a shortcut macro, if provided by the library) in a header
118  * file providing:
119  * - the specification type (a shortcut macro would omit exactly this)
120  * - the custom enum type and
121  * - information about the resource \alib{lang,Library} and the resource name for the custom type.
122  *
123  * And, of-course, the resources have to be defined.
124  *
125  * \anchor cpp_alib_lang_T_EnumMetaDataSpecification_exception_sample
126  * To stay with the example of class \alib{lang,Exception} above, let us look how the
127  * \alib formatter classes throw exceptions.
128  *
129  * First, the scoped enum class for the exceptions is defined:
130  *
131  * \snippet "strings/format/formatter.hpp" DOX_ALIB_ENUM_META_DATA_SPECIFCATION_using_enum
132  *
133  * Then, this enum is associated with exceptions. Of-course, we are using the shortcut macro
134  * provided by \e exception.hpp, instead of \ref ALIB_ENUM_SPECIFICATION "ALIB_ENUM_SPECIFICATION":
135  *
136  * \snippet "strings/format/formatter.hpp" DOX_ALIB_ENUM_META_DATA_SPECIFCATION_using_enum2
137  *
138  * In method \alib{lang,Library::init} of <b>%ALib Module Strings</b>, the resources for the
139  * exceptions are defined. Remember: besides the underlying integer value of the enum, class
140  * \b %Exception expects two strings: One with the name of the enum element, one with the
141  * resource name of the format string. In addition, we provide a pefix for the output of
142  * exception element names:
143  *
144  * \snippet "strings/stringlib.cpp" DOX_ALIB_ENUM_META_DATA_SPECIFCATION_using_enum3
145  *
146  * Now, \alib exceptions can be thrown by the formatter classes using their own enum types:
147  *
148  * \snippet "strings/format/formatterpythonstyle.cpp" DOX_ALIB_ENUM_META_DATA_SPECIFCATION_using_enum4
149  *
150  \~Comment ####################################################################################### \~
151  * ## Restrictions ##
152  * In respect to using the "core" macros and TMP struct specializations that
153  * \alib{lang,T_EnumMetaDataDecl,ALib Enum Meta Data} provides, there is one restriction: The
154  * specific meta data table of cutom types is to be defined using the resource system provided
155  * with \alib{lang,Library}.
156  *
157  * This restriction not a technical one, but a design decision: If also different provision methods
158  * for the data were allowed, the set of macros exposed would become much more complicated.
159  * With the openness of \alib resources, this should not be a problem. It is notable that
160  * its features include options to:
161  * - attach 3rd party resource sources and to
162  * - forbid external changes of the resources if needed.
163  *
164  * For the same reason of simplification, macro \ref ALIB_ENUM_SPECIFICATION includes
165  * macro \ref ALIB_LANG_ENUM_PRE_AND_POSTFIX providing fixed names for the resource strings,
166  * namely <c>"Prefix"</c> and <c>"Postfix"</c>.
167  *
168  \~Comment ####################################################################################### \~
169  * # Reference Documentation #
170  *
171  * @tparam TEMD Type provided by the library that exposes a definition of a specifc
172  * enum meta data tuple type.
173  */
