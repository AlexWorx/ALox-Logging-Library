37 /** ************************************************************************************************
38  * This class provides string based resources in a two-level, nested hash map. The outer map
39  * uses a category string key, while the inner uses another string key denoting the resource
40  * string name and hashing the resource string.<br>
41  *
42  * In addition to this simple hashing, with \alib module \ref ALIB_MODULE_CONFIGURATION in place,
43  * when loading resources, those are tried to be read from field #Config with priority.
44  * With the prioritization mechanics of class \alib{config,Configuration} itself, this
45  * allows for example to have language translations of resources in place, which - if only sparsely
46  * defined - automatically fallback through the different configuration plug-ins and finally get
47  * loaded from this classes internal simple nested hash map.
48  *
49  * This class is usually used with class \alib{lang,Library}, which also creates and distributes
50  * instances of this class to dependent libraries. In other words, normally instances of this class
51  * are never to be created by custom code.
52  *
53  * The documentation of class \alib{lang,Library} explains how a using software gets control over
54  * - the assignment of either the same or different instances of this class to different C++
55  * libraries
56  * - the initialization phases which allow to add custom resource "sources" on bootstrap.
57  *
58  * Due to the use of this class throughout \alib itself, almost all built-in names of
59  * \alib{config,Variable,configuration variables},
60  * \alib{lang,T_EnumMetaDataDecl,enum meta data},
61  * \alib{lang,Exception,exception messages},
62  * and so forth can be changed and customized.
63  *
64  *
65  * The simplest way to change names is by changing the values programatically (hard-coded) in
66  * the bootstrap section of a software.<br>
67  * A more advanced approach ist to use class \alib{config,IniFile}, load a resource file and
68  * add this to field \alib{lang,Resources::Config} as plug-in. Due to the "fallthrough" mechanism
69  * two things are possible:
70  * - An INI-file may be only "sparsely" filled. Only the values that are to be overwritten need
71  * to be set. For others, the default values apply.
72  * - More than one INI-file may be attached (with different priorities). This allows even to
73  * sort the supported languages by their preference.
74  *
75  * A fresh and complete INI-file that contains all default values can be created using method
76  * #FetchDefaults. Typically, a software would provide a special command line parameter to
77  * let a user export the resources once to an INI-file. The user then might choose to edit
78  * the file and copy it to a certain folder (and name scheme) that the application uses to search
79  * for (or a different command line parameter allows to use the file).
80  * If found, these files would be attached to the instance of this class with the next run of the
81  * software.
82  *
83  * The most advanced way to manage resources, is to write a custom configuration plug-in, to load
84  * resources from a place that the main software uses by default.
85  * Of-course, alternatively, such external sources might be used directly to fill the defaults -
86  * and this way leave field #Config untouched. There is a great freedom of choice.
87  **************************************************************************************************/
