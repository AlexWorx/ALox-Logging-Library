55 /**
56  * This namespace of \b %ALib holds classes that are providing an interface into
57  * multi threaded programming. The design of the classes tries to mimic the basic thread
58  * interface of the Java language.
59  *
60  * It is important to understand that
61  * (following \ref aworx::lib "the principle design goals of \b %ALib")
62  * multi threading support of \b %ALib is non intrusive in respect to multi-threading
63  * mechanisms of other C++ libraries and/or the underlying operating system.
64  * This means that a software entity that uses this library is not at all bound to use the thread
65  * classes found within this namespace.
66  *
67  * In contrast, using the classes found here, is of-course optional. \b %ALib usually
68  * provides a certain level of "source compatibility" between the programming languages
69  * C#, Java and C++.
70  *
71  * It is no problem, to have threads started using different methods and libraries than the
72  * one found here, as long that such threads become "native" operation system threads.
73  * If later such a thread uses method \alib{threads,Threads::CurrentThread}",
74  * a corresponding \alib{threads,Thread} object of \alib is created internally and returned.
75  * This way, the externally created thread is automatically "registered" with \b %ALib.
76  * If not stated differently in the detailed documentation, from this point in time,
77  * the thread can be interfaced with \b %ALib classes in the same manner as if it was created
78  * using them. The same or similar should be \c true for the opposite situation.
79  *
80  * Thread support of \b %ALib can be disabled, respectively enabled, using compiler symbols
81  * #ALIB_FEAT_THREADS_OFF and #ALIB_FEAT_THREADS_ON.
82  * When support is switched off, classes
83  * \alib{threads,Thread} and
84  * \alib{threads,Runnable} "Runnable" are not available. However, the classes
85  * \alib{threads,ThreadLockNR} "ThreadLockNR" and
86  * \alib{threads,ThreadLock} are.
87  * Their interface remains the same, but they do not perform any locking! Due to this behavior,
88  * it is possible to write code that implements due locking while still, in single threaded
89  * compilation, the code does not suffer any performance drawbacks, because lock code will be
90  * <em>pruned</em>.
91  *
92  */
