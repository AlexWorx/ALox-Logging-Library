54 /**
55  * This namespace of ALib holds classes that are providing an interface into
56  * multi threaded programming. The design of the classes tries to mimic the basic thread
57  * interface of the Java language.
58  *
59  * It is important to understand that
60  * (following \ref aworx::lib "the principle design goals of ALib")
61  * multi threading support of ALib is non intrusive in respect to multi-threading
62  * mechanisms of other C++ libraries and/or the underlying operating system.
63  * This means that a software entity that uses this library is not at all bound to use the thread
64  * classes found within this namespace.
65  *
66  * In contrast, using the classes found here, is of-course optional. ALib usually
67  * provides a certain level of "source compatibility" between the programming languages
68  * C#, Java and C++.
69  *
70  * It is no problem, to have threads started using different methods and libraries than the
71  * one found here, as long that such threads become "native" operation system threads.
72  * If later such a thread uses method
73  * \ref aworx::lib::threads::Thread::CurrentThread() "Thread::CurrentThread()",
74  * a corresponding \ref aworx::lib::threads::Thread "Thread" object of ALib
75  * is created internally and returned.
76  * This way, the externally created thread is automatically "registered" with ALib.
77  * If not stated differently in the detailed documentation, from this point in time,
78  * the thread can be interfaced with ALib classes in the same manner as if it was created
79  * using them. The same or similar should be \c true for the opposite situation.
80  *
81  * Thread support of ALib can be disabled, respectively enabled, using compiler symbols
82  * #ALIB_FEAT_THREADS_OFF and #ALIB_FEAT_THREADS_ON.
83  * When support is switched off, classes
84  * \ref aworx::lib::threads::Thread "Thread" and
85  * \ref aworx::lib::threads::Runnable "Runnable" are not available. However, the classes
86  * \ref aworx::lib::threads::ThreadLockNR "ThreadLockNR" and
87  * \ref aworx::lib::threads::ThreadLock "ThreadLock" are!
88  * Their interface remains the same, but they do not perform any locking. Due to this behavior,
89  * it is possible to write code that performs due locking while still, in single threaded
90  * compilation, the code does not suffer any performance drawbacks, because lock code will be
91  * <em>pruned</em>.
92  *
93  */
