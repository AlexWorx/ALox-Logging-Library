16 /** ************************************************************************************************
17  * This is the virtual base class to implement virtual methods invokable on supported
18  * boxed objects.
19  *
20  * The following gives a step by step guidance on how to create, attach and invoke virtual interface
21  * methods.
22  *
23  * <b>Defining a Custom %Interface:</b><br>
24  * To add a custom interface method to <b>%ALib %Boxing</b>, the following approach has to be taken:
25  * 1. A class needs to be derived from this class with a single custom virtual interface
26  * method named \c Invoke.<br>
27  * The first parameter of this virtual method needs to be of type <c>const aworx::Box&</c>. It
28  * will receive the boxed object that the interface method is invoked on. Other parameters
29  * and the return type may be chosen as needed.
30  * The method may be pure virtual.
31  * 2. From this class in turn, type-specific versions need to be derived, which implement
32  * method \b Invoke in a type specific way.
33  * 3. The type-specific versions should in addition be derived from template class
34  * \ref aworx::lib::lang::Singleton "Singleton". Alternatively, a singleton of themselves
35  * needs to be made accessible by other means.
36  * 4. The singletons of the type-specific classes need to be attached to the corresponding
37  * type-specific boxer using method
38  * \ref aworx::lib::boxing::Boxer::DefineInterface "Boxer::DefineInterface".<br>
39  * If, as proposed above, the type-specific classes use class \b %Singleton for implementing
40  * the singleton pattern, then template namespace function
41  * \ref aworx::lib::boxing::DefineInterface "DefineInterface" can be used for this purpose.
42  * 5. Optionally, one default version of the custom interface class may be derived. This
43  * would implement a default behavior for boxed types that have no type-specific version in
44  * place. Default interface singletons are announced using method
45  * \ref aworx::lib::boxing::Boxer::DefineDefaultInterface "Boxer::DefineDefaultInterface",
46  * or preferably with templated namespace function
47  * \ref aworx::lib::boxing::DefineDefaultInterface "DefineDefaultInterface":
48  *
49  * Steps 4 and 5 must be done only once, e.g. in the bootstrap section of the process.
50  *
51  * <b>Using a Custom Interface:</b><br>
52  * Now, the interface may be used as follows:
53  * 1. Template method
54  * \ref aworx::lib::boxing::Box::HasInterface "Box::HasInterface" is used to detect if an
55  * object of type \b %Box, supports a certain interface.
56  * As a template parameter to this method, the main interface class type has to be provided.
57  * (The one that is directly derived from this class \b %Interface and is the parent class
58  * to all type-specific specializations.)<br>
59  * Note that method \ref aworx::lib::boxing::Box::HasInterface "Box::HasInterface" will return
60  * \c false, if the boxed object does not support the interface, even if a default interface
61  * is available.
62  * 2. In the case that either \ref aworx::lib::boxing::Box::HasInterface "Box::HasInterface"
63  * returned \c true or a default interface implementation is available (which is expected to be
64  * globally known piece of information), template method
65  * \ref aworx::lib::boxing::Box::Invoke "Box::Invoke" can be used to invoke method \b %Invoke
66  * of the interface attached to the box. Likewise method \b Box::HasInterface, as a template
67  * parameter to this method, the main interface class type has to be provided.<br>
68  * The parameters passed to \b %Box::Invoke are the parameters defined of the interface
69  * method \b %Invoke (!), excluding the
70  * first one, which as described above needs to be <c>const aworx::Box&</c>.<br>
71  * If the return type of custom method \b Invoke is different to \c void, then this return type
72  * needs to be specified as the second template argument to method \b Box::Invoke.
73  *
74  * It is important to understand:
75  * - If a default interface of a certain type is set, then no check using method
76  * \b %Box::HasInterface is needed prior to invoking \b %Box::Invoke, because the latter
77  * will always find some invokable singleton. Furthermore, in this case, method
78  * \b %Box::HasInterface may be used to detect if a type-specific interface is in place.
79  * A typical invocation would look like this:
80  *
81  * aBox.Invoke<MyInterface>(5);
82  *
83  *
84  * - If \b no default interface of a certain type is set, then it is advised to use method
85  * \b %Box::HasInterface to check if an interface exists, prior to using
86  * \b %Box::Invoke. In debug compilations an assertion is
87  *
88  * The invocation would look like this:
89  *
90  * if( aBox.HasInterface<MyInterface>() )
91  * aBox.Invoke<MyInterface>(5);
92  *
93  * <b>Multiple Invocations:</b><br>
94  * In the case that an interface method is to be invoked several times in a row on the same
95  * instance \b %Box, the interface singleton might be received with method
96  * \ref aworx::lib::boxing::Box::GetInterface "Box::GetInterface" once and method \b %Invoke
97  * can then be directly invoked. In this case, parameter \p box has to be provided to the call, as
98  * demonstrated in this sample:
99  *
100  * MyInterface* mIF= aBox.GetInterface<MyInterface>();
101  * if( mIF != nullptr )
102  * for( int i= 0; i< 100; ++i )
103  * mIF->Invoke( aBox, i );
104  *
105  * Here, object \b aBox is first used to retrieve the interface and then it is passed to each
106  * invocation as a first parameter.
107  *
108  * \see
109  * For a step by step sample see chapter
110  * \ref alib_namespace_boxing_interfaces_sample "A Simple %Interface Sample" of the
111  * documentation of <b>%ALib %Boxing</b>.
112  **************************************************************************************************/
