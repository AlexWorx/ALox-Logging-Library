59 /** ************************************************************************************************
60  * Writes formatted text into an \ref aworx::lib::strings::AString "AString".
61  *
62  * \note
63  * The C++ version of this class relies heavily on classes found in module (namespace)
64  * \ref aworx::lib::boxing "ALib Boxing". In short, <b>ALib Boxing</b> allows to create methods
65  * that accept any type of argument as parameter, without the need of type conversion,
66  * wrapper classes, etc. Furthermore <b>ALib Boxing</b> allows to add virtual interfaces
67  * to C++ types, independent from the original implementation of the type itself. This means
68  * that any 3rd-party type can be adopted to support "boxing" and hence to be formatted with this
69  * class - without changing their implementation and with no access to their source code!<br>
70  * Knowledge about <b>ALib Boxing</b> is not needed to use this class with standard C++ types.
71  * Only if a string representation of custom types should be generated by this class with
72  * passing such custom types "as is", then support for <b>ALib Boxing</b> in respect to such
73  * custom types have to be added.
74  *
75  * \note
76  * The \ref aworx::lib::boxing "detailed documentation" of <b>ALib Boxing</b> describes how
77  * to do this. The "boxing interfaces" that are needed to be implemented for custom types
78  * to work with this formatters are
79  * \ref aworx::lib::strings::boxing::IApply "IApply" and optionally
80  * \ref aworx::lib::strings::boxing::IFormat "IFormat". The latter is used with
81  * \ref aworx::lib::strings::format::FormatterPythonStyle "FormatterPythonStyle", if a custom format
82  * specification is given for a replacement field.
83  *
84  * ##Different Supported Formatting Syntax##
85  *
86  * While this class is abstract, only derived classes that support a specific format syntax can
87  * be instantiated and used. Built-in formatters provided with \b %ALib are:
88  * - \ref aworx::lib::strings::format::FormatterPythonStyle "FormatterPythonStyle", which mimics
89  * the [Python formatting syntax](https://docs.python.org/3.5/library/string.html#format-string-syntax)
90  * and
91  * - \ref aworx::lib::strings::format::FormatterJavaStyle "FormatterJavaStyle", which mimics
92  * the formatting syntax of corresponding
93  * [Java class "Formatter"](http://docs.oracle.com/javase/8/docs/api/java/util/Formatter.html)
94  *
95  * One of the design goals of \b %ALib is to provide a library for (currently) three programming
96  * languages \b C++, \b Java and \b C#, which allows to reach a certain level of source code
97  * compatibility if used in cross-platform/language projects. Now, why do we provide
98  * <b>Python Style</b> instead of to mimic what is coming with \b .Net languages like \b C#?
99  *
100  * The answer is that we considered <b>Python Style</b> to be more powerful. It is also more powerful
101  * than the <b>Java Style</b> formatting syntax. But the latter is in its basics similar to
102  * good old \c printf format string style and hence for \b %ALib a must have.
103  * The good news is that in its very basics, <b>Python Style</b> is similar to \b .Net formatting.
104  * This way, there is some "familiar basic syntax" available for everybody.<br>
105  * Finally, it is worth to mention that the \b %ALib implementation of <b>Python Style</b> formatting
106  * is extending the original standard even a little.
107  *
108  *
109  * ##Concatenated Formatters And Concatenated Format Operations##
110  *
111  * In comparison to "usual", similar classes that are found in other libraries/languages, there
112  * are two main differences with this class:
113  * 1. This class supports the concatenation of formatters by passing an instance of
114  * a different derived type in the constructor. A formatter that has a (list of) concatenated
115  * other formatter(s) attached, allows to detect the provided format syntax by passing the format
116  * operation recursively to the concatenated formatter(s). Of-course this approach is limited to
117  * such syntax definitions that are not ambiguous. (...It works well with formatters
118  * \b %FormatterPythonStyle and \b %FormatterJavaStyle!)
119  *
120  * 2. Usually (in other libraries) the interface of formatting methods accept a dedicated
121  * format string and list of arguments to format. Method #Format of this class however
122  * expects just an argument list. The processing then is done as follows:
123  * The first argument in the list is tested to be a format string. If it is, method #format
124  * will report how many arguments form the list have been "consumed" and formatting will continue
125  * with the first argument after the last consumed one. If the actual argument is not identified
126  * as a format string (tested with each concatenated formatter), the string is just appended to
127  * the target and formatting continues with the next argument in the list.<br>
128  * If at the end of this processing loop a single argument is left, then this is not checked
129  * for being a format string. Instead, it will be just appended to the target as is.
130  * An argument that potentially could be a format string is simply skipped (ignored) if it
131  * evaluates to \c nullptr.
132  *
133  * This behaviour has the following consequences:
134  * - No format string needs to be passed. In this case, the arguments are appended to the target
135  * \b %AString in their sequence order
136  * - More than one format string (with according) arguments might be passed.
137  * - The different format strings might even follow a different syntax.
138  *
139  * The biggest \b advantage of this implementation is that format strings and arguments might be
140  * collected prior to invoking the formatter. Collection might happen at different places and times
141  * in a process with the only restriction that not more or less arguments might be added to the
142  * overall list than referred to in the format string. In other words, format strings have to
143  * be well set in respect to the arguments they consume.
144  *
145  * The biggest <b>potential issue</b> of this approach is that a string argument that is not meant
146  * to be a format string might be accidentally interpreted as one if it happens to contain one or
147  * more placeholder characters that are matched by one of the concatenated formatters. (In this case it
148  * it is likely that the format is invalid and an error is written. In debug mode, even an assertion
149  * might be raised). To avoid such ambiguity, the following considerations have to be taken into
150  * account when passing arguments:
151  * - Ambiguities might only happen with string-type arguments. Other types that are converted to
152  * string representations are never considered a potential format string.
153  * - Strings that contain a newline-character (ascii \b 0x0a) are not considered to be a format string.
154  * (Newline characters in a format string have to be escaped using a formatter-specific notation,
155  * e.g. \c "\n").
156  * - A string-type argument that is the last in the list (or the only one) is safe
157  * (no ambiguity can occur) because the last argument is never checked for being a format string.
158  * - %String-type arguments that do not contain placeholder characters defined by one of the formatters
159  * are safe.
160  *
161  * \attention
162  * <b>The rule that results from this is:</b><br>
163  * If none of the conditions mentioned above can be guaranteed, then the argument in question
164  * should be passed together with a preceding simple formatting string, for example,
165  * \c "{}" for \b %FormatterPythonStyle or
166  * \c "%s" in case of \b %FormatterJavaStyle.
167  *
168  * ##Choosing The %Formatter##
169  * As elaborated above, there are two built-in formatter types and those can be concatenated.
170  * Finally, there is the option to implement an own, custom formatter. For the undecided, the
171  * recommendation is to use <b>Java Style</b> for simple, quick <c>printf-</c> like formatting
172  * and to choose <b>Python Style</b> in the moment formatting requirements get more complex.
173  * A special case and good sample for the use of \b ALib formatting features is found in the
174  * [ALox Logging Library](https://github.com/AlexWorx/ALox-Logging-Library). Here the special
175  * situation is that \b %ALox itself is a library and hence exposes its formatters used to
176  * users. Therefore \b %ALox (by default) uses a <b>Python Style</b> formatter and a concatenated
177  * <b>Java Style</b> version. This way, simple samples will work from scratch for users of all
178  * three languages!
179  *
180  * ##Usage##
181  * The class is abstract, hence only derived types can be instantiated.
182  * Those types are constructed with optionally providing a concatenated instance of another
183  * derived type. The optional concatenated formatter object has a lower priority in respect to
184  * detecting a format string.
185  *
186  * Two overloaded methods \b Format exists:
187  * - The first method accepts a target \b AString and a variadic list of arguments. This method
188  * can be used if the format string(s) and all arguments are available/known at the point of
189  * invocation.
190  * - The second method accepts a target \b AString and a list of arguments. This method
191  * can be used if format string(s) and arguments are collected prior to the final invocation
192  * of the formatter.
193  *
194  * \note
195  * The use of this class is not "thread safe".
196  *
197  * ##Implementing A Custom %Formatter##
198  * To implement a custom formatter, the abstract methods found in this class have to be implemented.
199  * With class
200  * \ref aworx::lib::strings::format::FormatterStdImpl "FormatterStdImpl" a - still abstract -
201  * descendant class is provided. This class implements method #format but introduces a set of
202  * new abstract methods to overwrite. Although, those are more methods, these methods are probably
203  * more simple to implement and therefore deriving from this class might be considered for new
204  * custom types.
205  *
206  * The two built-in <b>Formatter</b>s provided with \b %ALib,
207  * \ref aworx::lib::strings::format::FormatterPythonStyle "FormatterPythonStyle" and
208  * \ref aworx::lib::strings::format::FormatterJavaStyle "FormatterJavaStyle" derive from
209  * \b %FormatterStdImpl and copying the source code of one of them might be a good start for
210  * implementing a custom type.
211  **************************************************************************************************/
