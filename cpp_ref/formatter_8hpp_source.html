190 /** ************************************************************************************************
191  * Writes formatted text into an \ref aworx::lib::strings::AString "AString".
192  *
193  * \note
194  * The C++ version of this class relies heavily on classes found in module (namespace)
195  * \ref aworx::lib::boxing "ALib Boxing". In short, <b>%ALib %Boxing</b> allows to create methods
196  * that accept any type of argument as parameter, without the need of type conversion,
197  * wrapper classes, etc. Furthermore <b>%ALib %Boxing</b> allows to add virtual interfaces
198  * to C++ types, independent from the original implementation of the type itself. This means
199  * that any 3rd-party type can be adopted to support "boxing" and hence to be formatted with this
200  * class - without changing their implementation and with no access to their source code!<br>
201  * Knowledge about <b>%ALib %Boxing</b> is not needed to use this class with standard C++ types.
202  * Only if a string representation of custom types should be generated by this class with
203  * passing such custom types "as is", then support for <b>%ALib %Boxing</b> in respect to such
204  * custom types have to be added.
205  *
206  * \note
207  * The \ref aworx::lib::boxing "detailed documentation" of <b>%ALib %Boxing</b> describes how
208  * to do this. The "boxing interfaces" that are needed to be implemented for custom types
209  * to work with this formatters are
210  * \ref aworx::lib::strings::boxing::IApply "IApply" and optionally
211  * \ref aworx::lib::strings::boxing::IFormat "IFormat". The latter is used with
212  * \ref aworx::lib::strings::format::FormatterPythonStyle "FormatterPythonStyle", if a custom format
213  * specification is given for a replacement field.
214  *
215  * ##Different Supported Formatting Syntax##
216  *
217  * While this class is abstract, only derived classes that support a specific format syntax can
218  * be instantiated and used. Built-in formatters provided with \b %ALib are:
219  * - \ref aworx::lib::strings::format::FormatterPythonStyle "FormatterPythonStyle", which mimics
220  * the [Python formatting syntax](https://docs.python.org/3.5/library/string.html#format-string-syntax)
221  * and
222  * - \ref aworx::lib::strings::format::FormatterJavaStyle "FormatterJavaStyle", which mimics
223  * the formatting syntax of corresponding
224  * [Java class "Formatter"](http://docs.oracle.com/javase/8/docs/api/java/util/Formatter.html)
225  *
226  * One of the design goals of \b %ALib is to provide a library for (currently) three programming
227  * languages \b C++, \b Java and \b C#, which allows to reach a certain level of source code
228  * compatibility if used in cross-platform/language projects. Now, why do we provide
229  * <b>Python Style</b> instead of to mimic what is coming with \b .Net languages like \b C#?
230  *
231  * The answer is that we considered <b>Python Style</b> to be more powerful. It is also more powerful
232  * than the <b>Java Style</b> formatting syntax. But the latter is in its basics similar to
233  * good old \c printf format string style and hence for \b %ALib a must have.
234  * The good news is that in its very basics, <b>Python Style</b> is similar to \b .Net formatting.
235  * This way, there is some "familiar basic syntax" available for everybody.<br>
236  * Finally, it is worth to mention that the \b %ALib implementation of <b>Python Style</b> formatting
237  * is extending the original standard even a little.
238  *
239  *
240  * ##Concatenated Formatters And Concatenated Format Operations##
241  *
242  * In comparison to "usual", similar classes that are found in other libraries/languages, there
243  * are two main differences with this class:
244  * 1. This class supports the concatenation of formatters by passing an instance of
245  * a different derived type in the constructor. A formatter that has a (list of) concatenated
246  * other formatter(s) attached, allows to detect the provided format syntax by passing the format
247  * operation recursively to the concatenated formatter(s). Of-course this approach is limited to
248  * such syntax definitions that are not ambiguous. (...It works well with formatters
249  * \b %FormatterPythonStyle and \b %FormatterJavaStyle!)
250  *
251  * 2. Usually (in other libraries) the interface of formatting methods accept a dedicated
252  * format string and list of arguments to format. Method #Format of this class however
253  * expects just an argument list. The processing then is done as follows:
254  * The first argument in the list is tested to be a format string. If it is, method #format
255  * will report how many arguments form the list have been "consumed" and formatting will continue
256  * with the first argument after the last consumed one. If the actual argument is not identified
257  * as a format string (tested with each concatenated formatter), the string is just appended to
258  * the target and formatting continues with the next argument in the list.<br>
259  * If at the end of this processing loop a single argument is left, then this is not checked
260  * for being a format string. Instead, it will be just appended to the target as is.
261  * An argument that potentially could be a format string is simply skipped (ignored) if it
262  * evaluates to \c nullptr.
263  *
264  * This behaviour has the following consequences:
265  * - No format string needs to be passed. In this case, the arguments are appended to the target
266  * \b %AString in their sequence order
267  * - More than one format string (with according) arguments might be passed.
268  * - The different format strings might even follow a different syntax.
269  *
270  * The biggest \b advantage of this implementation is that format strings and arguments might be
271  * collected prior to invoking the formatter. Collection might happen at different places and times
272  * in a process with the only restriction that not more or less arguments might be added to the
273  * overall list than referred to in the format string. In other words, format strings have to
274  * be well set in respect to the arguments they consume.
275  *
276  * The biggest <b>potential issue</b> of this approach is that a string argument that is not meant
277  * to be a format string might be accidentally interpreted as one if it happens to contain one or
278  * more placeholder characters that are matched by one of the concatenated formatters. (In this case it
279  * it is likely that the format is invalid and an error is written. In debug mode, even an assertion
280  * might be raised). To avoid such ambiguity, the following considerations have to be taken into
281  * account when passing arguments:
282  * - Ambiguities might only happen with string-type arguments. Other types that are converted to
283  * string representations are never considered a potential format string.
284  * - Strings that contain a newline-character (ascii \b 0x0a) are not considered to be a format string.
285  * (Newline characters in a format string have to be escaped using a formatter-specific notation,
286  * e.g. \c "\n").
287  * - A string-type argument that is the last in the list (or the only one) is safe
288  * (no ambiguity can occur) because the last argument is never checked for being a format string.
289  * - %String-type arguments that do not contain placeholder characters defined by one of the formatters
290  * are safe.
291  *
292  * \attention
293  * <b>The rule that results from this is:</b><br>
294  * If none of the conditions mentioned above can be guaranteed, then the argument in question
295  * should be passed together with a preceding simple formatting string, for example,
296  * \c "{}" for \b %FormatterPythonStyle or
297  * \c "%s" in case of \b %FormatterJavaStyle.
298  *
299  * ##Choosing The %Formatter##
300  * As elaborated above, there are two built-in formatter types and those can be concatenated.
301  * Finally, there is the option to implement an own, custom formatter. For the undecided, the
302  * recommendation is to use <b>Java Style</b> for simple, quick <c>printf-</c> like formatting
303  * and to choose <b>Python Style</b> in the moment formatting requirements get more complex.
304  * A special case and good sample for the use of \b %ALib formatting features is found in the
305  * [ALox Logging Library](https://github.com/AlexWorx/ALox-Logging-Library). Here the special
306  * situation is that \b %ALox itself is a library and hence exposes its formatters used to
307  * users. Therefore \b %ALox (by default) uses a <b>Python Style</b> formatter and a concatenated
308  * <b>Java Style</b> version. This way, simple samples will work from scratch for users of all
309  * three languages!
310  *
311  * ##Usage##
312  * The class is abstract, hence only derived types can be instantiated.
313  * Those types are constructed with optionally providing a concatenated instance of another
314  * derived type. The optional concatenated formatter object has a lower priority in respect to
315  * detecting a format string.
316  *
317  * Two overloaded methods \b Format exists:
318  * - The first method accepts a target \b AString and a variadic list of arguments. This method
319  * can be used if the format string(s) and all arguments are available/known at the point of
320  * invocation.
321  * - The second method accepts a target \b AString and a list of arguments. This method
322  * can be used if format string(s) and arguments are collected prior to the final invocation
323  * of the formatter.
324  *
325  * \note
326  * The use of this class is not "thread safe".
327  *
328  * ##Implementing A Custom %Formatter##
329  * To implement a custom formatter, the abstract methods found in this class have to be implemented.
330  * With class
331  * \ref aworx::lib::strings::format::FormatterStdImpl "FormatterStdImpl" a - still abstract -
332  * descendant class is provided. This class implements method #format but introduces a set of
333  * new abstract methods to overwrite. Although, those are more methods, these methods are probably
334  * more simple to implement and therefore deriving from this class might be considered for new
335  * custom types.
336  *
337  * The two built-in <b>Formatter</b>s provided with \b %ALib,
338  * \ref aworx::lib::strings::format::FormatterPythonStyle "FormatterPythonStyle" and
339  * \ref aworx::lib::strings::format::FormatterJavaStyle "FormatterJavaStyle" derive from
340  * \b %FormatterStdImpl and copying the source code of one of them might be a good start for
341  * implementing a custom type.
342  **************************************************************************************************/
