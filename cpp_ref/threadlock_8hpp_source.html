35 /** ************************************************************************************************
36  * This class allows *mutual exclusive access* to resources shared by different threads.
37  * In other words, access to certain data that is accessed by different threads, can
38  * be protected if each thread (aka critical section code) uses the same ThreadLock to control the
39  * access to such data.
40  *
41  * If an Acquire is not followed by a corresponding Release(), other threads will not be able to
42  * gain control to this object and will wait endlessly (deadlock situation).
43  * When the Acquire and Release invocations happen within the same code block, then it is
44  * highly recommended to use a stack instantiation of class ALIB::Owner to acquire and release
45  * objects of this class.
46  *
47  * ThreadLock uses an internal counter to allow multiple calls to Acquire() and to be freed only
48  * when a same amount of Release() calls are performed. This behavior can be switched off by a
49  * constructor parameter. If switched off, each recursive Acquire() call will not be counted and
50  * each call to Release() will instantly free the mutex. This mode is not very recommended,
51  * the standard use is recursive mode.
52  *
53  * Furthermore, ThreadLock allows to disable locking using setUnsafe(). The objective here is to
54  * gain execution speed, as thread synchronization causes relatively expensive system calls.
55  * Nevertheless, it is sometimes obvious that the same code may run in a thread safe mode
56  * one time and without thread locking the next time. Therefore, for performance critical
57  * code, it is quite useful to be able to control this behavior.
58  **************************************************************************************************/
