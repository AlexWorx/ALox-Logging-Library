19 /** ********************************************************************************************
20  * @name Warning-, Error- and Assertion- Reports for Debug Compilations
21  *
22  * The macros listed here are defined in two different versions, depending on the ALib distribution.
23  * With single <b>ALib Modules</b> that do not incorporate class
24  * \ref aworx::lib::lang::Report and family, namespace function
25  * \ref aworx::lib::lang::DbgSimpleALibMsg (and overloads) are used to write the messages.
26  * The default implementation of this message invokes \b %assert(false) if the message is of
27  * error type.<br>
28  * There is a very simple "plug-in" concept in place that allows to redirect this method to
29  * a user defined one which may act differently.
30  *
31  * With the complete ALib library distribution two things happen:
32  * - the aforementioned plug-in is implemented and an ALib Report is generated inside
33  * the plug-in function.
34  * - The macros themselves are redefined to directly use the ALib Report facility.
35  * Such redefinition happens "in the middle" of header inclusion (as early as possible), with
36  * the effect that most classes of ALib are compiled using the simple version, but still use
37  * the mechanics of ALib Reports. Other parts of ALib classes, and of-course all user code,
38  * will have the more sophisticated, report-based versions of the macros in place. The
39  * huge advantage of them is, that those accept an arbitrary list of arguments which are boxed
40  * and passed to a formatter. This allows to easily compose detailed and formatted debug messages.
41  *
42  * \note It is advised for users of complete ALib to rather use the
43  * [ALox Logging Library](http://alexworx.github.io/ALox-Logging-Library/index.html) instead
44  * of these macros for debug output. ALib uses these macros internally for the sole reason
45  * to be lean in respect module deployment and to avoid cyclic dependencies with \b %ALox.
46  *
47  *
48  * \def ALIB_ERROR
49  * Writes the given message objects as an error.
50  *
51  * \def ALIB_WARNING
52  * Writes the given message objects as an warning.
53  *
54  * \def ALIB_MESSAGE
55  * Writes the given message.
56  *
57  * \def ALIB_ASSERT
58  * If given condition is \c false, error message "Assertion Failed" is written.
59  *
60  * \def ALIB_ASSERT_ERROR
61  * If given condition is \c false, given message objects are written as an error.
62  *
63  * \def ALIB_ASSERT_WARNING
64  * If given condition is \c false, given message objects are written as a warning.
65  *
66  * @}
67  */

206 /** ************************************************************************************************
207  * \name Macros for One time warnings
208  *
209  * This group of macros enable the implementation of "one-time warnings", which
210  * are warnings that are issued only once per debug session.
211 
212  * The warnings are executed using
213  * \ref aworx::lib::lang::Report::DoReport "Report::DoReport" (on the \b Report instance received by
214  * \ref aworx::lib::lang::Report::GetDefault "Report::GetDefault")
215  * and are effective only in debug compilations. They are pruned in release code
216  * (dependent on symbol #ALIB_DEBUG).
217  *
218  * <b> Declaration of an one-time warning </b><p>
219  * If an one-time warning is declared per type using \ref ALIB_WARN_ONCE_PER_TYPE_DECL only one
220  * warning for all objects will occur and the warnings can be enabled or disabled only for all
221  * instances of a class together.
222  *
223  * In contrast to this, declaring an one-time warning per instance using
224  * \ref ALIB_WARN_ONCE_PER_INSTANCE_DECL, a warning will occur once for each object.
225  *
226  * Both macros have to be placed in the public member section of a class. In addition to that,
227  * per type declarations need to be completed with a definition using macro
228  * \ref ALIB_WARN_ONCE_PER_TYPE_DEFINE in the implementation file of the type.
229  *
230  * <b> Raise Warnings </b><p>
231  * Within the implementation of a class that has OTW-declarations, the macros
232  * #ALIB_WARN_ONCE, #ALIB_WARN_ONCE_IF and #ALIB_WARN_ONCE_IF_NOT is used to issue the warning.
233  *
234  * <b> Controlling the warnings</b>
235  * The using code of a class that has OTW-declarations can enable and disable (or re-enable)
236  * one-time warnings using macros
237  * - \ref ALIB_WARN_ONCE_PER_TYPE_ENABLE,
238  * - \ref ALIB_WARN_ONCE_PER_TYPE_DISABLE,
239  * - \ref ALIB_WARN_ONCE_PER_INSTANCE_ENABLE and
240  * - \ref ALIB_WARN_ONCE_PER_INSTANCE_DISABLE.
241  *
242  * For example, if a certain warning or condition seems right for a specific situation,
243  * an one-time warning can be disabled and re-enabled after that situation. Or if it is a per
244  * instance warning, it can be disabled after creating that instance.
245  *
246  * <b> Parameters of the macros</b>
247  * The following parameters are expected by the macros:
248  *
249  * - <em>libdecl:</em><br>
250  * Must match a macro that defines the import/export declaration of the source code entity
251  * (see \ref ALIB_API for a sample of how this symbol is declared in ALib itself).
252  * - <em>identifier</em><br>
253  * Identifies the one-time warning. Has to apply to the rules of C++ variable names and can
254  * other wise be chosen freely.
255  * - <em>className</em><br>
256  * Names the class when defining or setting a per class one-time warnings.
257  * - <em>instance</em><br>
258  * Names a reference to the instance when using per instance one-time warnings.
259  * Within a method of the corresponding type, use \c *this.
260  * - <em>default</em><br>
261  * The default value when declaring, respectively implementing the variable. Can be \c true for
262  * enabled by default, or \c false for disabled.
263  **************************************************************************************************/
