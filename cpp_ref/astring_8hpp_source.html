51 /** ********************************************************************************************
52  * This is a template struct which answers the question
53  * "Can an object (value) of type \p TApplicable be passed to method \ref AString::Apply?
54  * By default, the answer is no. In particular, the default implementation of the struct inherits
55  * \c std::false_type.
56  *
57  * To make a custom type "applicable", this struct needs to be specialized for the custom type
58  * with inheriting from \c std::true_type and with implementing method #Apply.
59  *
60  * This concept of specializing this struct "once" and then being able to pass custom types
61  * to objects of class
62  * \ref aworx::lib::strings::AString "AString" has a central role for using \b %ALib in a
63  * \ref aworx::lib "non intrusive" way: Methods of \b %ALib classes (or third-party code that uses
64  * \b %ALib) may offer APIs that can be fed with custom types without the prior need of conversion.
65  *
66  * A list of built-in specializations for standard C++ and \b %ALib types is given and documented
67  * \ref alib_namespace_strings_astring_application_vs_interface "here".
68  *
69  * To restrict the need for specializations of this method to a single variant of the external type,
70  * template type \p TApplicable to specialize will always be a the plain custom type. No pointer,
71  * const, etc. versions need to be implemented.
72  *
73  * The implementation of method #Apply may do 'whatever' is wanted with the %AString provided in
74  * parameter \p target. For string types, simple append operations are predefined. More
75  * complex operations format the string, convert integer and floating point numbers, etc.
76  *
77  * For basic user defined string types that get adopted to \b %ALib string system using a specialization
78  * of struct
79  * \ref aworx::lib::strings::T_String "T_String", no specialization of this struct is needed,
80  * because method \ref aworx::lib::strings::AString::Apply "AString::Apply" will detect that and
81  * append the custom string.
82  *
83  * Built-in specializations of this function for certain types exist, as listed in reference
84  * documentation of class \ref aworx::lib::strings::AString "AString".
85  * Some types exists only for the reason to be "applicable" to <b>%AString</b>!
86  * As a sample, class
87  * \ref aworx::lib::strings::Format::Field "Format::Field" allows to align strings
88  * when they are applied. The invocation is simple
89  * and nicely readable:
90  * \snippet "DOX_ALIB_APPLYTO.cpp" DOX_ALIB_APPLYTO_FIELD
91  * which produces:
92  * \snippet "DOX_ALIB_APPLYTO_FIELD.txt" OUTPUT
93  *
94  * The next sample implements this function for \b %ALib class
95  * \ref aworx::lib::time::Ticks "Ticks" to print out a formatted date:
96  * \snippet "DOX_ALIB_APPLYTO.cpp" DOX_ALIB_APPLYTO_DEFINITION
97  * With this definition included, user code might now do this:
98  * \snippet "DOX_ALIB_APPLYTO.cpp" DOX_ALIB_APPLYTO_USE
99  *
100  * The output will be something like this:
101  *
102  * \snippet "DOX_ALIB_APPLYTO.txt" OUTPUT
103  *
104  * \note This sample is kept simple. In real life, class
105  * \ref aworx::lib::time::TicksCalendarTime "TicksCalendarTime" should be used to
106  * implement more options.
107  **********************************************************************************************/
108 template<typename TApplicable> struct T_Apply : public std::false_type
109 {
110  /** ********************************************************************************************
111  * Default implementation of Apply. This method must never be invoked and therefore
112  * throws an assertion if it is.
113  * Prior to do so, a \c static_assert will prevent code to apply types to an
114  * object of class \b %AString that are not applicable. However, it might happen that this
115  * compile-time assertion fails to be activated in some tricky situations.
116  * In this case, the runtime assert will step in.
117  *
118  * Implementations for specialized versions of this struct might do whatever they are supposed
119  * to with \p target. Actions are not restricted to appending something.
120  *
121  * \note
122  * Some more complex applicable types might want to return extended information by setting
123  * own members during the application process. To achieve this, the corresponding
124  * implementations of this method have to cast the given const reference to a
125  * non-const reference. This is considered a good practice, as the const specifier
126  * most probably was added by the TMP operations of method
127  * \ref aworx::lib::strings::AString::Apply "AString::Apply" anyhow - only for the reason
128  * to have one single template specialization per type.
129  *
130  * @param target The object of type %AString that will have object \p src applied.
131  * @param src The source object of arbitrary type will get applied to %AString \p target.
132  * @return The number of characters that got appended should be returned
133  * ('should' here means that at least a non zero, positive value has to be returned)
134  * and -1 if the object \p src which is to be applied represented a \e nulled object.
135  * Returning -1 causes the AString to be \e nulled if the apply operation was invoked through
136  * \ref aworx::lib::strings::AString::operator= "AString assignment operator".<br>
137  * If nothing is was appended, this method must return 0.
138  **********************************************************************************************/
139  static inline
140  integer Apply( AString& target, const TApplicable& src )
141  {
142  (void) target;
143  (void) src;
144 
145  ALIB_WARNINGS_START_TMP
146 
147  // prevent invoking us with unknown types
148  using TPlain= typename std::remove_const <
149  typename std::remove_pointer <
150  typename std::remove_reference<
151  typename std::remove_const < TApplicable >::type>::type>::type>::type;
152  using TAElem= typename std::remove_extent<TPlain>::type;
153 
154  static_assert(
155  std::is_arithmetic < TPlain>::value
156  || std::is_base_of <String, TPlain>::value
157  || std::is_same <decltype(nullptr), TPlain>::value
158 
159  || (( std::is_same <char, TAElem>::value
160  || std::is_same <wchar_t, TAElem>::value
161  || std::is_same <char16_t, TAElem>::value
162  || std::is_same <char32_t, TAElem>::value
163  ) && std::is_array <TPlain> ::value )
164 
165  || T_String <TPlain >::value
166  || T_Apply <TPlain >::value
167  , "ALib: T is not a known type to append to AString. Implement struct T_Apply<T> to enable conversion.");
168 
169  //ALIB_TMP_SHOW_TYPE_IN_DEBUGGER(TPlain)
170  //assert(0);
171 
172  ALIB_DEBUG_CODE( appendErrorToAString( target ) );
173 
174  ALIB_WARNING("Unknown type for T_ApplyTo()");
175 
176  return 0;
177 
178  ALIB_WARNINGS_RESTORE
179  }

262 /** ************************************************************************************************
263  * Specializes class
264  * \ref aworx::lib::strings::TString "TString" to implement mutable, non-constant
265  * strings using writable buffer memory.
266  *
267  * There are two possible types of buffers:
268  * - <b>Internal buffers</b><br>
269  * This is the standard case and implements a buffer that is allocated from
270  * <em>free memory</em> (aka 'the heap') that eventually grows over time and never shrinks, unless
271  * explicitly demanded by the user of the class. Those buffers are deleted when objects
272  * of this type are deleted.
273  * - <b>External buffers</b><br>
274  * Set with method #SetBuffer. These buffers don't get deleted. However, if their capacity
275  * is exceeded, they will get automatically replaced by an internal buffer.
276  * (See field #ReplaceExternalBuffer to control warnings on that in debug compilation).
277  *
278  *<b>Construction/Destruction</b><p>
279  * The following ways of construction are provided:
280  * - <b>Default construction</b><br>
281  * Does not allocate a character buffer. The allocation is this way deferred to
282  * the first operation that writes data into the object. Alternatively, an external buffer
283  * can be set before such operation is invoked.
284  * - <b>Construction with initial capacity</b><br>
285  * With constructor #AString(size_t), an initial length can be provided. This construction
286  * variant should be used, if the maximum length the object that will be reached during its
287  * lifetime is predictable, because it avoids iterative, increasing buffer allocations that
288  * otherwise would be performed until that size is reached once.
289  * Furthermore there are constructors
290  * - <b>Construction with external buffers</b><br>
291  * With constructor #AString(char*, integer), an external buffer is set. This constructor is
292  * protected and therefore accessible by derived classes' constructors only.
293  * This is to avoid an otherwise likely common misunderstanding that this constructor would
294  * copy the contents of a provided cstring. For users of this class, the alternative to this
295  * constructor is using a combination of the default constructor and method #SetBuffer which
296  * has exactly the same effect.
297  * - <b>Construction with strings</b><br>
298  * Constructors #AString(const TApplicable&), AString(const String&, index, integer=) and copy constructor
299  * #AString(const AString&) will allocate an internal buffer of the capacity needed
300  * and copy the contents of the given string type.
301  *
302  * Destruction will free internally allocated buffers.
303  * \note Method #SetBuffer provides a boolean parameter that also allows to let an object of this
304  * class take control on the provided buffer, namely delete it in the destructor. In this
305  * case, a buffer set with this method is considered an internal buffer, rather than an
306  * external one.
307  *
308  * \anchor alib_namespace_strings_astring_move
309  *<b>Move constructor and move assignment</b><p>
310  * This class provides a (noexcept) move constructor which grabs the buffer of a given movable
311  * \b %AString, as long as this given object does not use an external buffer. In the latter case, the
312  * contents is copied like in the copy constructor.<br>
313  * This class does not provide a move assignment operator. The rational for this is that normally
314  * a "more temporary" AString would be assigned to a "less temporary" one. In this case, it would
315  * not be helpful to replace the allocated storage, each time.
316  *
317  * \note
318  * In general, while assignment of other objects of type \b %AString is possible through
319  * #operator=(const AString&), the advised code style is to use a combination of #Clear and
320  * append \ref _ "_()" (e.g. <em>myString._()._(otherString)</em>) to do the assignment instead.
321  * This leads to exactly the same result as long as the \b %AString that is assigned is not \e nulled.<br>
322  * \note
323  * In the case that it is \e nulled, using the assignment operator will set this object also
324  * to \e nulled state, hence dispose any currently allocated buffer. This, for most cases is not
325  * what is wanted.
326  *
327  * \anchor alib_namespace_strings_astring_write_access
328  * <b>Write Access to the Buffer</b><p>
329  * Parent class \ref aworx::lib::strings::String "String" holds its protected field
330  * \ref aworx::lib::strings::String::buffer "String::buffer" in an anonymous C++ union of two pointers,
331  * one typed <em>const char*</em> and the other <em>char*</em>.
332  * This class exposes the non-constant buffer pointer of that union with method #VBuffer.
333  * This allows users of this class to <em>freely</em> operate on the buffer with any <em>C</em>
334  * or <em>C++</em> algorithm available. Of-course, it is up to the programmer doing this to
335  * keep invariants that this class naturally constitutes intact.
336  *
337  * In addition to this, a bunch of methods allow the modification of characters.
338  * #operator[] is returning a reference to a char in contrast to the same operator defined
339  * in class \b %String. This allows assignments of values when using an object with this operator as
340  * an lvalue.
341  *
342  * \anchor alib_namespace_strings_astring_nullable
343  * <b>Nullable AStrings</b><p>
344  * As discussed in the namespace documentation (\ref alib_namespace_strings_nullable)
345  * parent class
346  * \ref aworx::lib::strings::String "String" provides an interface to detect \e nulled objects.
347  * This concept is duly implemented in this class. An object of type #AString is \e nulled
348  * when no internal buffer is allocated or external buffer is set.
349  *
350  * If default constructed, constructed with zero size, \c nullptr or any other string type
351  * object which is \e nulled, no buffer is created and hence the new object is in \e nulled
352  * state and inherited method
353  * aworx::lib::strings::String::IsNull "IsNull" will return \c true for that object.
354  * Consequently, it makes a difference if an \b %AString is constructed using <em>AString()</em> or
355  * <em>AString(\"\")</em>.
356  * This allows to differentiate between \e nulled AStrings and empty \b %AStrings, which is quite handy
357  * in certain situations. An object that was filled already can be reset (nulled) by
358  * either assigning a \c nullptr, by invoking \ref SetBuffer "SetBuffer(0)" or by invoking
359  * #SetNull on the instance. (See also methods #IsNull, #IsNotNull and #Capacity.)
360  * The methods #Equals, #CompareTo and the overloaded comparison #operator== and
361  * #operator!= allow nullptr comparisons. e.g. a \e nulled \b %AString equals to another
362  * \e nulled \b %AString but not to an empty but not \e nulled AString.
363  *
364  * To make this more clear, note the following <b>sample code</b> which does not throw an assertion:
365  * \snippet "DOX_ALIB_ASTRING.cpp" DOX_ALIB_ASTRING_NULLED
366  *
367  * \note
368  * - As mentioned above, to avoid loosing an allocated buffer, use a combination of
369  * #Clear and append \ref _ "_()" to perform assignments with other string types instead of
370  * using one of the assignment operators (#operator=). No check for \e nulled string
371  * is needed when appending them to an object of this type.
372  *
373  * - Even if an \b %AString object is \e nulled, inherited method #ToCString will return a
374  * (zero terminated) valid empty <em>char*</em>.
375  * This has the advantage that in many situations the nulled state is not
376  * needed to be handled (for those cases where the difference between a \e nulled and an empty
377  * string is irrelevant).
378  *
379  * \anchor alib_namespace_strings_astring_applyto
380  * <b>Applying Objects to AStrings</b><p>
381  * The concept of appending things to a string is generalized in the C++
382  * implementation of this class and named <em>applying</em> instead. While applying string
383  * types is the same as appending string types in other language versions of \b %ALib, the
384  * '<em>application</em>' of non-string types allows flexible and well readable code design.
385  *
386  * The central mechanism used to apply arbitrary types is similar to what is used by class
387  * \ref aworx::lib::strings::String "String" and the partially specialized template struct
388  * \ref aworx::lib::strings::T_String "T_String".
389  * By specializing template struct
390  * \ref aworx::lib::strings::T_Apply "T_Apply"
391  * it is possible to apply objects of 'external', user defined types to objects of this class.
392  *
393  * These partly specialized templates are used by this classes' method #Apply. Because this
394  * method, although declared <em>public</em>, has a more internal character, in standard
395  * situations its invocation is performed indirectly through methods:
396  * - Constructor #AString(const TApplicable&)
397  * - Operator operator <<(const T&)
398  * - Method #_(const T&) (which is a synonym for <em>operator << </em> but allowing non-checked invocations)
399  * - Operator operator=(const T&)
400  *
401  * For more information refer to documentation of partially specialized template struct
402  * \ref aworx::lib::strings::T_Apply "T_Apply" and
403  *
404  * \anchor alib_namespace_strings_astring_application_vs_interface
405  * <b>Application Instead of Explicit Interface Methods</b><p>
406  * Various methods that exist in the implementations of class \b %AString in different programming
407  * languages versions of \b %ALib, are missing in this C++ implementation.
408  * Here, they have been replaced by the concept of <em>applying</em> types.
409  * The following provides a list of predefined applicable types found in \b %ALib:
410  *
411  * - Character types \c char, \c wchar_t, \c char16_t or \c char32_t, and strings of those
412  * (pointers to the types or arrays of them) will be appended.<br>
413  * Wide characters and wide character strings <em>([const] wchar_t*)</em> will be converted to
414  * encoded multi-byte character strings according to the currently set locale. (See
415  * \ref aworx::lib::ALIB::Init "ALIB::Init" for more information about setting locale and
416  * \ref aworx::lib::strings::String::ToWString "String::ToWString" for converting in the other
417  * direction.)
418  * - C++ fundamental arithmetic types (e.g. \c float, \c int, <c>signed char</c>, etc.) will be
419  * converted to string representations using static object
420  * \ref aworx::lib::strings::NumberFormat::Computational "NumberFormat::Computational" and appended.
421  * - Type \c bool appends the string literal \c "true" respectively \c "false".
422  * - Class \ref aworx::lib::strings::Format "Format", with public inner classes
423  * - \ref aworx::lib::strings::Format::Field "Format::Field"
424  * - \ref aworx::lib::strings::Format::Tab "Format::Tab"
425  * - \ref aworx::lib::strings::Format::Bin "Format::Bin"
426  * - \ref aworx::lib::strings::Format::Hex "Format::Hex"
427  * - \ref aworx::lib::strings::Format::Oct "Format::Oct"
428  * - Class \ref aworx::lib::boxing::Box "Box" by invoking boxing interface
429  * \ref aworx::lib::strings::boxing::IApply "IApply".
430  * - Some support is given for third party types. See \ref aworx::lib::strings::thirdparty
431  * for details.
432  *
433  * <b>Insert, Replace and Delete</b><p>
434  * The following insert, delete and replace operations are provided:
435  * - #InsertAt To insert a string at an arbitrary position.
436  * - #InsertChars To insert a quantity of characters at an arbitrary position.
437  * - \ref ReplaceSubstring(const String&,integer,integer) "ReplaceSubstring" To replace a substring with a new string .
438  * - \ref ReplaceRegion(char,integer,integer) "ReplaceRegion" To replace a substring with a given character.
439  * - #SearchAndReplace to search all occurrences of a substring and replace by a different string.
440  * - #Delete To delete an arbitrary substring.
441  * - #DeleteStart To delete n characters from the start.
442  * - #DeleteEnd To delete n characters from the end.
443  *
444  **************************************************************************************************/

758  /** ****************************************************************************************
759  * Resizes the internal buffer to meet exactly the given size.
760  *
761  * The following rules apply:
762  * - The string represented by this instance is copied to the new buffer.
763  * If this is larger than the new buffer size, the string is cut at the end to fit.
764  * - If the desired new size is 0, then the currently allocated buffer will be released
765  * and the objects state is \e nulled.

771  * replace the current buffer is when the current buffers' life-cycle is (already)
772  * internally managed and it has the same size than what is requested.

904  "Previous allocation request was too short" );

915  * Increases the allocation size at least by the value provided.
916  *

920  {

922 
923  // big enough?
924  ALIB_ASSERT_WARNING ( actCapacity < length + minimumGrowth,
925  "Unnecessary invocation of Grow()" );

935  integer newCapacity= actCapacity + (actCapacity / 2);
936  if ( newCapacity < length + minimumGrowth )
937  newCapacity+= minimumGrowth;
938 
939  if ( newCapacity < 16 )
940  newCapacity= 16;
941 
942  SetBuffer( newCapacity );
943  }

1621  /** ****************************************************************************************
1622  * This generic method allows to apply an object of custom type \p TApplicable.
1623  * For most types, 'apply' means to append a string representation of the given object.<br>
1624  * In detail, the implementation of this method proceeds as follows:
1625  * - If a pointer type is detected and value is nullptr, nothing is done and -1 is returned.
1626  * This allows the caller to react on nullptr values (e.g. the assignment operator,
1627  * would set this object to \e nulled state if -1 was returned).
1628  * - If a character type (\c char, \c wchar_t, \c char16_t or \c char32_t) is passed,
1629  * it is appended.

1631  * string is appended. %String types detected are:

1634  * - \c char16_t*, \c char16_t[]
1635  * - \c char32_t*, \c char32_t[]

1637  * - Custom types that have a specialization of template struct
1638  * \ref T_String.
1639  * - If above does not match, it is checked for a specialization of template struct

1641  * are removed and if \p TApplicable is a pointer type, even this is ignored.

1655  * there might be a few types not fetched by this static assert.

1660  * through:

1679  * its \e nulled state. In other words, \e nulled strings keep being \e nulled if empty objects

1682  * to improve performance in critical code sections.
