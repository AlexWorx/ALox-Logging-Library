ALib Macros

Detailed Description

The preprocessor macros listed here are utility macros used by A-Worx Library ALib and are documented as they might be useful for users of ALib same as ALib itself.

The definition of the macros is dependent on ALib Symbols To Select Code Fragments, which in turn mostly depend on ALib Compilation Symbols.

Not listed here, are macros of ALox. Those are documented with

  • ALox Macros For Debug Logging
  • ALox Macros For Release Logging

ALib Version And Feature Verification

Macros used to verify that the ALib binary used has the right version is is compiled with the same feature set as the current compilation unit. For this, version and feature flags are compiled statically into the library. The macros are used internally, respectively 'automatically'. Hence, they need not to be used directly clients of the library.

#define ALIB_REVISION   0

Macros Supporting ALib Enum Types

The macros in this section allow to specialize TMP structs T_EnumMetaDataDecl, T_EnumIsBitwise and T_EnumIsIteratable for scoped or non-scoped enum types.

In respect to ALib enum meta data, besides the general macro, two predefined, oftenly used meta data types (tuple types) are provided: one which is just adding element names for output of enum element names and a the second that also allows parsing the names.
In addition, for both predefined meta data types a version which standardizes the externalization of the data with class Resources is provided.

Attention
All macros in this section have to be placed in global scope (outside of namespaces and types).

Macro used to declare enumeration type TEnum to denote ALib Configuration Variables. In particular, the macro includes the following code:

  • Specializes TMP struct T_EnumMetaDataDecl for type TEnum and associate type VariableDecl::TTuple for its meta data.
  • Passes parameters ResourceLibrary and ResourceName to macro ALIB_LANG_RESOURCED.
  • Specializes method EnumMetaData::CheckLoad to use a detail method for loading the meta data from the resources.

With this - and the corresponding resource data! - in place, elements of TEnum can be used to declare configuration variables by passing them to one of following constructors and methods:

  • VariableDecl::VariableDecl(TEnum)
  • Variable::Variable(TEnum)
  • Variable::Variable(TEnum,const StringTypes&...)
  • Variable::Declare(TEnum)
  • Variable::Declare(TEnum,const StringTypes&...)

The resource data is a table of values separated by character '|' ("pipe" character), which has to contain the eight values of tuple type VariableDecl::TTuple. Elements 3 ("DefaultValue") and 7 ("Comments"), in the resources are interpreted as a resource name. This way, both values are loaded from separated resource strings. This has the following advantages:

  • The values may contain the separation character '|'.
  • The values can be manipulated within the resources more easily.

Note that for loading the resources, static template method ResourcedTupleLoader::LoadTable is used. As documented, this method (optionally) allows to separate each variable declaration entry into an own, numbered resource string.

See also
Namespace documentation aworx::lib::config for information on configuration data and variables.

Used to associate a custom enum type with a type of enum meta data that is in turn associated to given type TSpecific. Hence, this macro is used by code to enable its custom enum types to be usable with a library that accepts enum elements with a certain meta data scheme.

Along with the specialization of T_EnumMetaDataSpecification, the macro also comprises macros

  • ALIB_LANG_RESOURCED,
  • ALIB_LANG_ENUM_PRE_AND_POSTFIX (using fixed strings "Prefix" and "Postfix" as resource names, and
  • ALIB_LANG_ENUM_READ_WRITE.

In addition, EnumMetaData::CheckLoad is specialized for type TEnum to use method T_EnumMetaDataSpecification::LoadTable which has to be implemented using macro ALIB_ENUM_SPECIFICATION_IMPL.

Often, libraries provide an explicit shortcut to this macro, that omits parameter TSpecific and gives a more readable name. For examples, see ALIB_LANG_EXCEPTIONS or ALIB_CONFIG_VARIABLES.

This preprocessor macro defines an object of class Owner. The special thing about it is, that with using this macro there is no need to "invent" an (otherwise unreferenced) identifier for that definition. Basically, this macro exists, because C++ does not support anonymous local instances.
As a sample, without using this macros a piece of code code using classes Owner/ Ownable could look like this:

     Owner myOwner( myOwnable );

With the use of this macro, the code changes to:

{

    ...
