41 /** ************************************************************************************************
42  * Implements a \ref aworx::lib::strings::format::Formatter "Formatter" according to the
43  * [formatting standards of the Python language](https://docs.python.org/3.5/library/string.html#format-string-syntax).
44  *
45  * \note
46  * Inherited, public fields of parent class \b FormatterStdImpl provide important possibilities
47  * for changing the formatting behavior of instances of this class. Therefore, do not forget
48  * to consult the \ref aworx::lib::strings::format::FormatterStdImpl "parent classes documentation".
49  *
50  * In general, the original \b Python specification is covered quite well. However, there are
51  * some differences, some things are not possible (considering python being a scripting language)
52  * but then there are also found some very helpful extensions to that standard. Instead of repeating
53  * a complete documentation, please refer to the
54  * [Python Documentation](https://docs.python.org/3.5/library/string.html#format-string-syntax)
55  * as the foundation and then take note of the following list of differences, extensions and
56  * general hints:
57  *
58  * - <b>General Notes:</b>
59  * \b Python defines a placeholder field as follows
60  *
61  * "{" [field_name] ["!" conversion] [":" format_spec] "}"
62  *
63  *
64  * - This formatter is <b>less strict</b> in respect to the order of the format symbols. E.g.
65  * it allows <c>{:11.5,}</c> where Python allows only <c>{:11,.5}</c>
66  *
67  * - With this class being derived from
68  * \ref aworx::lib::strings::format::FormatterStdImpl "FormatterStdImpl", features of the parent are
69  * available to this formatter as well. This is especially true and sometimes useful in respect to
70  * setting default values number formatting. For example, this allows to modify all number output
71  * without explicitly repeating the settings in each placeholder of format strings. Other options
72  * for example the grouping characters used with hexadecimal numbers, can not be even changed
73  * with the <b>Python Style</b> formatting options. The only way of doing so is modifying the
74  * properties of the formatter object prior to the format operation.
75  *
76  * - Nested replacements in format specification fields are (by nature of this implementation
77  * language) \b not supported.
78  *
79  * <p>
80  * - <b>Positional arguments and field name:</b>
81  * - By the nature of the implementation language (<em>C++, no introspection</em>) of this class,
82  * \b field_name can \b not be the name of an identifier, an attribute name or an array element
83  * index. It can only be a positional argument index, hence a number that chooses a different
84  * index in the provided argument list.<br>
85  * However, the use of field names is often a requirement in use cases that offer configurable
86  * format string setup to the "end user". Therefore, there are two alternatives to cope
87  * with the limitation:
88  * - In simple cases, it is possible to just add all optionally needed data in the argument list,
89  * document their index position and let the user use positional argument notation to choose
90  * the right value from the list.
91  * - More elegant however, is the use of class
92  * \ref aworx::lib::strings::format::PropertyFormatter "PropertyFormatter"
93  * which extends the format specification by custom identifiers which control the placement
94  * of corresponding data in the format argument list. This class uses a translator table from
95  * identifier strings to custom callback functions. This way, much more than just simple
96  * field names are allowed.
97  *
98  * - When using positional arguments in a format string placeholders, the Python formatter
99  * implementation does not allow to switch from <b>automatic field indexing</b> to explicit
100  * indexing. This \b %Aib implementation does allow it. The automatic index (aka no positional
101  * argument is given for a next placeholder) always starts with index \c 0 and is incremented
102  * each time automatic indexing is used. Occurrences of explict indexing have no influence
103  * on the automatic indexing.
104  *
105  *
106  * <p>
107  * - <b>Binary, Hexadecimal and Octal Numbers:</b>
108  * - Binary, hexadecimal and octal output is <b>cut in size</b> (!) when a field width is given that
109  * is smaller than the resulting amount of digits of the number arguments provided.
110  * \note This implies that a value written might not be equal to the value given.
111  * This is not a bug but a design decision. The rational behind this is that with this
112  * behavior, there is no need to mask lower digits when passing the arguments to the
113  * format invocation. In other words, the formatter "assumes" that the given field width
114  * indicates that only a corresponding number of lower digits are of interest.
115  *
116  * - The number <b>grouping option</b> (<c>','</c>) can also be used with binary, hexadecimal and octal
117  * output.
118  * The types support different grouping separators for nibbles, bytes, 16-bit and 32-bit words.
119  * Changing the separator symbols, is not possible with the format fields of the format strings
120  * (if it was, this would become very incompatible to Python standards). Changes have to be made
121  * prior to the format operation by modifying field
122  * \ref aworx::lib::strings::format::FormatterStdImpl::AlternativeNumberFormat "AlternativeNumberFormat"
123  * which is provided through parent class \b %FormatterStdImpl.
124  *
125  * - Alternative form (\c '#') adds prefixes as specified in members
126  * - \ref aworx::lib::strings::NumberFormat::BinLiteralPrefix "BinLiteralPrefix",
127  * - \ref aworx::lib::strings::NumberFormat::HexLiteralPrefix "HexLiteralPrefix" and
128  * - \ref aworx::lib::strings::NumberFormat::OctLiteralPrefix "OctLiteralPrefix".
129  *
130  * For upper case formats, those are taken from field
131  * \ref aworx::lib::strings::format::FormatterStdImpl::DefaultNumberFormat "DefaultNumberFormat",
132  * for lower case formats from
133  * \ref aworx::lib::strings::format::FormatterStdImpl::AlternativeNumberFormat "AlternativeNumberFormat".
134  * However, in a alignment with the \b Python specification, \b both default to lower case
135  * literals \c "0b", \c "0o" and \c "0x". All defaults may be changed by the user.
136  *
137  *
138  * <p>
139  * - <b>Floating point values:</b>
140  * - If floating point values are provided without a type specification in the format string, then
141  * all values of
142  * \ref aworx::lib::strings::format::FormatterStdImpl::DefaultNumberFormat "DefaultNumberFormat"
143  * are used to format the number
144  * - For lower case floating point format types (\c 'f' and \c 'e'), the values specified in
145  * attributes \b %ExponentSeparator, \b %NANLiteral and \b %INFLiteral of object
146  * \ref aworx::lib::strings::format::FormatterStdImpl::AlternativeNumberFormat "AlternativeNumberFormat"
147  * are used. For upper case types (\c 'F' and \c 'E') the corresponding attributes in
148  * \ref aworx::lib::strings::format::FormatterStdImpl::DefaultNumberFormat "DefaultNumberFormat" apply.
149  * - Fixed point formats (\c 'f' and 'F' types) are not supported to use arbitrary length.
150  * See class \ref aworx::lib::strings::NumberFormat "NumberFormat" for the limits.
151  * Also, very high values and values close to zero may be converted to scientific format.
152  * Finally, if flag
153  * \ref aworx::lib::strings::NumberFormat::ForceScientific "ForceScientific"
154  * of field #DefaultNumberFormat is \c true, types \c 'f' and 'F' behave like types \c 'e' and 'E'.
155  * - When both, a \p width and a \p precision is given, then the \p precision determines the
156  * fractional part, even if the type is \b 'g' or \b 'G'. This is different than specified with
157  * Python formatter, which uses \p precision as the overall width in case of types
158  * \b 'g' or \b 'G'.
159  * - The 'general format' type for floats, specified with \c 'g' or \c 'G' in the python
160  * implementation limits the precision of the fractional part, even if \p precision is not
161  * further specified. This implementation does limit the precision only if type is \c 'f'
162  * or \c 'F'.
163  *
164  * <p>
165  * - <b>%String Conversion:</b><br>
166  * If \e type \c 's' (or no \e type) is given in the \b format_spec of the replacement field,
167  * a string representation of the given argument is used.
168  * In \b Java and \b C# such representation is received by invoking <c>Object.[t|T]oString()</c>.
169  * Consequently, to support string representations of custom types, in these languages
170  * the corresponding <b>[t|T]oString()</b> methods of the type have to be implemented.
171  *
172  * In C++ the arguments are "boxed" into objects of type
173  * \ref aworx::lib::boxing::Box "Box". For the string representation, the formatter invokes
174  * virtual boxing interface
175  * \ref aworx::lib::strings::boxing::IApply "IApply". A default implementation exists which
176  * for custom types appends the type name and the memory address of the object in hexadecimal
177  * format. To support custom string representations (for custom types), this boxing interface
178  * needs to be implemented for the type in question. Information and sample code on how to do this
179  * is found in the documentation of <b>%ALib %Boxing</b>, chapter
180  * \ref alib_namespace_boxing_strings_iapply "6.3 Interface IApply".
181  *
182  * - <b>Hash-Value Output:</b><br>
183  * In extension (and deviation) of the Python specification, format specification type \c 'h' and
184  * its upper case version \c 'H' is implemented. The hash-values of the argument object is
185  * written in hexadecimal format. Options of the type are identical to those of \c 'x',
186  * respectively \c 'X'.
187  *
188  * In the C++ language implementation of \b %ALib, instead of hash-values of objects, the result
189  * of method \ref aworx::lib::boxing::Box::UnboxRaw "Box::UnboxRaw" is printed. In case of boxed
190  * class-types (that use default boxing mechanics), this will show the memory address of the
191  * given instance.
192  *
193  * - <b>Boolean output:</b><br>
194  * In extension (and deviation) of the Python specification, format specification type \c 'B'
195  * is implemented. The word \b "true" is written if the given value represents a boolean \c true
196  * value, \b "false" otherwise.
197  *
198  * In the C++ language implementation of \b %ALib, the argument is \c true if
199  * the result of method \ref aworx::lib::boxing::Box::UnboxRaw "Box::UnboxRaw" is not \c 0.
200  *
201  * <p>
202  * - <b>%Custom %Format Specifications:</b><br>
203  * With \c Python formatting syntax, placeholders have the the following syntax:
204  *
205  * "{" [field_name] ["!" conversion] [":" format_spec] "}"
206  *
207  * The part that follows the colon is called \b format_spec. \b Python passes this portion
208  * of the placeholder to a built-in function \c format(). Now, each type may interpret
209  * this string in a type specific way. But most built-in \b Python types do it along what
210  * they call the ["Format Specification Mini Language"](https://docs.python.org/3.5/library/string.html#format-specification-mini-language).
211  *
212  * With this implementation, the approach is very similar. The only difference is that the
213  * "Format Specification Mini Language" is implemented for standard types right within this class.
214  * But before processing \b format_spec, this class will check if an argument provides a custom
215  * implementation for formatting.
216  *
217  * <br>In \b C++, again an <b>%ALib %Boxing</b> interface class is used.
218  * If a given boxed argument provides boxing interface
219  * \ref aworx::lib::strings::boxing::IFormat "IFormat", this interface is invoked and string
220  * \b format_spec is passed for custom processing.
221  *
222  * Information and sample code on how to adopt custom types to support this interface is
223  * found in the documentation of <b>%ALib %Boxing</b>, chapter
224  * \ref alib_namespace_boxing_strings_iformat "6.4 Interface IFormat".
225  *
226  * Built-in class
227  * \ref aworx::lib::time::Ticks "Ticks" supports custom formatting with boxing interface
228  * \ref aworx::lib::time::IFormat_TTicks "IFormat_TTicks"
229  *
230  *
231  * <p>
232  * - <b>Conversions:</b><br>
233  * In the \b Python placeholder syntax specification:
234  *
235  * "{" [field_name] ["!" conversion] [":" format_spec] "}"
236  *
237  * symbol \c '!' if used prior to the colon <c>':'</c> defines
238  * what is called the <b>conversion</b>. With \b Python, three options are given:
239  * \c '!s' which calls \c str() on the value, \c '!r' which calls \c repr() and \c '!a' which
240  * calls \c ascii(). This is of-course not applicable to this formatter. As a replacement,
241  * this class extends the original specification of that conversion using \c '!'.
242  * The following provides a list of conversions supported. The names given can be abbreviated
243  * at any point and ignore letter case, e.g. \c !Upper can be \c !UP or just \c !u.
244  * In addition, multiple conversions can be given by concatenating them, each repeating
245  * character \c '!'.<br>
246  * The conversions supported are:
247  *
248  * - <b>!Upper</b><br>
249  * Converts the contents of the field to upper case.
250  *
251  * - <b>!Lower</b><br>
252  * Converts the contents of the field to lower case.
253  *
254  * - <b>!Quote</b><br>
255  * Puts quote characters \c '"' around the field.
256  * Note that these characters are not respecting any optional given field width but instead
257  * are added to such.<br>
258  * An alias name for \!Quote is given with \b !Str. As the alias can be abbreviated to \b !s,
259  * this provides compatibility with the \b Python specification.
260  *
261  * - <b>!ESC[<|>]</b><br>
262  * In its default behavior or if \c '<' is specified, certain characters are converted to escape
263  * sequences.
264  * If \c '>' is given, escape sequences are converted to their (ascii) value.
265  * See \ref aworx::lib::strings::Format::Escape "Format::Escape" for details about the conversion
266  * that is performed.<br>
267  * An alias name for \b !ESC< is given with \b !a which provides compatibility
268  * with the \b Python specification.
269  * \note If \b !ESC< is used in combination with \b !Quote, then \b !ESC< should be the first
270  * conversion specifier. Otherwise, the quotes inserted might be escaped as well.
271  *
272  * - <b>!Fill[Cc]</b><br>
273  * Inserts as many characters as denoted by the integer type argument.
274  * By default the fill character is space <c>' '</c>. It can be changed with optional character
275  * 'C' plus the character wanted.
276  *
277  * - <b>!Tab[Cc][NNN]</b><br>
278  * Inserts fill characters to extend the length of the string to be a multiple of a tab width.
279  * By default the fill character is space <c>' '</c>. It can be changed with optional character
280  * 'C' plus the character wanted. The tab width defaults to \c 8. It can be changed by adding
281  * an unsigned decimal number.
282  *
283  * - <b>!ATab[[Cc][NNN]|Reset]</b><br>
284  * Inserts an "automatic tabulator stop". These are tabulator positions that are stored
285  * internally and are automatically extended in the moment the actual contents exceeds the
286  * currently stored tab-position. An arbitrary amount of auto tab stop and field width
287  * (see <b>!AWith</b> below) values is maintained by the formatter.
288  *
289  * Which each new invocation of \alib{strings::format,Formatter::Format}, the first auto value
290  * is chosen and with each use of \c !ATab or \c !AWidth, the next value is used.<br>
291  * By default, the fill character is space <c>' '</c>. It can be changed with optional character
292  * 'C' plus the character wanted. The optional number provided gives the growth value by which
293  * the tab will grow if its position is exceeded. This value defaults to \c 3.
294  * The positions currently stored with the formatter can be reset with providing argument
295  * \c Reset. Alternatively to this, outside of a formatting process, the tab stops can be reset
296  * by invoking method \alib{strings::format,Formatter::Reset}.
297  *
298  * Both, auto tab and auto width conversions may be used to increase readability of multiple
299  * output lines. Of-course, output is not completely tabular, only if those values that result
300  * in the biggest sizes are formatted first. If a perfect tabular output is desired, the data
301  * to be formatted may be processed twice: Once to temporary buffer which is disposed and then
302  * a second time to the desired output stream.
303  *
304  * \note Method \alib{strings::format,Formatter::ReleaseDefault} resets the auto size values
305  * in case no multiple (recursive) acquirements had been performed.
306  *
307  * - <b>!AWidth[NNN|Reset]</b><br>
308  * Increases field width with repetitive invocations of format whenever a field value did not
309  * fit to the actually stored width. Optional decimal number \b NNN is added as a padding value.
310  * for more information, see <b>!ATab</b> above.
311  *
312  * - <b>!Xtinguish</b><br>
313  * Does not print anything. This is useful if format strings are externalized, e.g defined
314  * in \alib{lang,Library::Res,library resources}. Modifications of such resources might use this
315  * conversion to suppress the display of arguments (which usually are hard-coded).
316  *
317  * - <b>!Replace<search><replace></b><br>
318  * Searches string \p search and replaces with \p replace. Both values have to be given
319  * enclosed by characters \c ’<' and \c ’>'. In the special case that \p search is empty
320  * (<c><></c>), string \p replace will be inserted if the field argument is an empty string.
321  *
322 \~Comment ####################################################################################### \~
323  * # Reference Documentation #
324  * @throws aworx::lib::strings::format::Exceptions
325  * - \alib{strings::format::Exceptions,ArgumentIndexOutOfBounds}
326  * - \alib{strings::format::Exceptions,IncompatibleTypeCode}
327  * - \alib{strings::format::Exceptions,MissingClosingBracket}
328  * - \alib{strings::format::Exceptions,MissingPrecisionValuePS}
329  * - \alib{strings::format::Exceptions,DuplicateTypeCode}
330  * - \alib{strings::format::Exceptions,UnknownTypeCode}

332  * - \alib{strings::format::Exceptions,UnknownConversionPS}
333  * - \alib{strings::format::Exceptions,PrecisionSpecificationWithInteger}
334  **************************************************************************************************/
