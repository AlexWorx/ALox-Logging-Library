41 /** ************************************************************************************************
42  * Implements a \ref aworx::lib::strings::Formatter "Formatter" according to the
43  * [formatting standards of the Python language](https://docs.python.org/3.5/library/string.html#format-string-syntax).
44  *
45  * \note
46  * Inherited, public fields of parent class \b FormatterStdImpl provide important possibilities
47  * for changing the formatting behavior of instances of this class. Therefore, do not forget
48  * to consult the \ref aworx::lib::strings::FormatterStdImpl "parent classes documentation".
49  *
50  * In general, the original \b Python specification is covered quite well. However, there are
51  * some differences, some things are not possible (considering python being a scripting language)
52  * but then there are also found some very helpful extensions to that standard. Instead of repeating
53  * a complete documentation, please refer to the
54  * [Python Documentation](https://docs.python.org/3.5/library/string.html#format-string-syntax) as the foundation
55  * and then take note of the following list of differences, extensions and general hints:
56  *
57  * - <b>General Notes:</b>
58  * \b Python defines a placeholder field as follows
59  *
60  * "{" [field_name] ["!" conversion] [":" format_spec] "}"
61  *
62  * - By the nature of the implementation language of this class, \b field_name can not be the
63  * name of an identifier, an attribute name or an array element index.
64  * It can only be a positional argument index.
65  *
66  * - Python formatter does not allow to switch from <b>automatic field indexing</b> to explicit
67  * indexing. This implementation does allow it. The automatic index always starts with index
68  * \c 0 and is incremented each time automatic indexing is used.
69  * Occurrences of explict indexing have no influence on the automatic indexing.
70  *
71  *
72  * - This formatter is <b>less strict</b> in respect to the order of the format symbols. E.g.
73  * it allows <c>{:11.5,}</c> where Python allows only <c>{:11,.5}</c>
74  *
75  * - With this class being derived from
76  * \ref aworx::lib::strings::FormatterStdImpl "FormatterStdImpl", features of the parent are
77  * available to this formatter as well. This is especially true and sometimes useful in respect to
78  * setting default values number formatting. For example, this allows to modify all number output
79  * without explicitly repeating the settings in each placeholder of format strings. Other options
80  * for example the grouping characters used with hexadecimal numbers, can not be even changed
81  * with the <b>Python Style</b> formatting options. The only way of doing so is modifying the
82  * properties of the formatter object prior to the format operation.
83  *
84  * - Nested replacements in format specification fields are (by nature of this implementation
85  * language) \b not supported.
86  *
87  * <p>
88  * - <b>Binary, Hexadecimal and Octal Numbers:</b>
89  * Binary, hexadecimal and octal output is <b>cut in size</b> (!) when a field width is given that
90  * is smaller than the resulting amount of digits of the number arguments provided.
91  * \note This implies that a value written might not be equal to the value given.
92  * This is not a bug but a design decision. The rational behind this is that with this
93  * behavior, there is no need to mask lower digits when passing the arguments to the
94  * format invocation. In other words, the formatter "assumes" that the given field width
95  * indicates that only a corresponding number of lower digits are of interest.
96  *
97  * - The number <b>grouping option</b> (<c>','</c>) can also be used with binary, hexadecimal and octal
98  * output.
99  * The types support different grouping separators for nibbles, bytes, 16-bit and 32-bit words.
100  * Changing the separator symbols, is not possible with the format fields of the format strings
101  * (if it was, this would become very incompatible to Python standards). Changes have to be made
102  * prior to the format operation by modifying field
103  * \ref aworx::lib::strings::FormatterStdImpl::AlternativeNumberFormat "AlternativeNumberFormat"
104  * which is provided through parent class \b %FormatterStdImpl.
105  *
106  * - Alternative form (\c '#') adds prefixes as specified in members
107  * - \ref aworx::lib::strings::NumberFormat::BinLiteralPrefix "BinLiteralPrefix",
108  * - \ref aworx::lib::strings::NumberFormat::HexLiteralPrefix "HexLiteralPrefix" and
109  * - \ref aworx::lib::strings::NumberFormat::OctLiteralPrefix "OctLiteralPrefix".
110  *
111  * For upper case formats, those are taken from field
112  * \ref aworx::lib::strings::FormatterStdImpl::DefaultNumberFormat "DefaultNumberFormat",
113  * for lower case formats from
114  * \ref aworx::lib::strings::FormatterStdImpl::AlternativeNumberFormat "AlternativeNumberFormat".
115  * However, in a alignment with the \b Python specification, \b both default to lower case
116  * literals \c "0b", \c "0o" and \c "0x". All defaults may be changed by the user.
117  *
118  *
119  * <p>
120  * - <b>Floating point values:</b>
121  * - If floating point values are provided without a type specification in the format string, then
122  * all values of
123  * \ref aworx::lib::strings::FormatterStdImpl::DefaultNumberFormat "DefaultNumberFormat"
124  * are used to format the number
125  * - For lower case floating point format types (\c 'f' and \c 'e'), the values specified in
126  * attributes \b %ExponentSeparator, \b %NANLiteral and \b %INFLiteral of object
127  * \ref aworx::lib::strings::FormatterStdImpl::AlternativeNumberFormat "AlternativeNumberFormat"
128  * are used. For upper case types (\c 'F' and \c 'E') the corresponding attributes in
129  * \ref aworx::lib::strings::FormatterStdImpl::DefaultNumberFormat "DefaultNumberFormat" apply.
130  * - Fixed point formats (\c 'f' and 'F' types) are not supported to use arbitrary length.
131  * See class \ref aworx::lib::strings::NumberFormat "NumberFormat" for the limits.
132  * Also, very high values and values close to zero may be converted to scientific format.
133  * Finally, if flag
134  * \ref aworx::lib::strings::NumberFormat::ForceScientific "ForceScientific"
135  * of field #DefaultNumberFormat is \c true, types \c 'f' and 'F' behave like types \c 'e' and 'E'.
136  * - When both, a \p width and a \p precision is given, then the \p precision determines the
137  * fractional part, even if the type is \b 'g' or \b 'G'. This is different than specified with
138  * Python formatter, which uses \p precision as the overall width in case of types
139  * \b 'g' or \b 'G'.
140  * - The 'general format' type for floats, specified with \c 'g' or \c 'G' in the python
141  * implementation limits the precision of the fractional part, even if \p precision is not
142  * further specified. This implementation does limit the precision only if type is \c 'f'
143  * or \c 'F'.
144  *
145  * <p>
146  * - <b>%String Conversion:</b><br>
147  * If \e type \c 's' (or no \e type) is given in the \b format_spec of the replacement field,
148  * a string representation of the given argument is used.
149  * In \b Java and \b C# such representation is received by invoking <c>Object.[t|T]oString()</c>.
150  * Consequently, to support string representations of custom types, in these languages
151  * the corresponding <b>[t|T]oString()</b> methods of the type have to be implemented.
152  *
153  * In C++ the arguments are "boxed" into objects of type
154  * \ref aworx::lib::boxing::Box "Box". For the string representation, the formatter invokes
155  * virtual boxing interface
156  * \ref aworx::lib::strings::boxing::IApply "IApply". A default implementation exists which
157  * for custom types appends the type name and the memory address of the object in hexadecimal
158  * format. To support custom string representations (for custom types), this boxing interface
159  * needs to be implemented for the type in question. Information and sample code on how to do this
160  * is found in the documentation of <b>ALib Boxing</b>, chapter
161  * \ref alib_namespace_boxing_strings_iapply "6.3 Interface IApply".
162  *
163  * - <b>Hash-Value Output:</b><br>
164  * In extension (and deviation) of the Python specification, format specification type \c 'h' and
165  * its upper case version \c 'H' is implemented. The hash-values of the argument object is
166  * written in hexadecimal format. Options of the type are identical to those of \c 'x',
167  * respectively \c 'X'.
168  *
169  * In the C++ language implementation of \b %ALib, instead of hash-values of objects, the result
170  * of method \ref aworx::lib::boxing::Box::UnboxRaw "Box::UnboxRaw" is printed. In case of boxed
171  * class-types (that use default boxing mechanics), this will show the memory address of the
172  * given instance.
173  *
174  * - <b>Boolean output:</b><br>
175  * In extension (and deviation) of the Python specification, format specification type \c 'B'
176  * is implemented. The word \b "true" is written if the given value represents a boolean \c true
177  * value, \b "false" otherwise.
178  *
179  * In the C++ language implementation of \b %ALib, the argument is \c true if
180  * the result of method \ref aworx::lib::boxing::Box::UnboxRaw "Box::UnboxRaw" is not \c 0.
181  *
182  * <p>
183  * - <b>%Custom %Format Specifications:</b><br>
184  * With \c Python formatting syntax, placeholders have the the following syntax:
185  *
186  * "{" [field_name] ["!" conversion] [":" format_spec] "}"
187  *
188  * The part that follows the colon is called \b format_spec. \b Python passes this portion
189  * of the placeholder to a built-in function \c format(). Now, each type may interpret
190  * this string in a type specific way. But most built-in \b Python types do it along what
191  * they call the ["Format Specification Mini Language"](https://docs.python.org/3.5/library/string.html#format-specification-mini-language).
192  *
193  * With this implementation, the approach is very similar. The only difference is that the
194  * "Format Specification Mini Language" is implemented for standard types right within this class.
195  * But before processing \b format_spec, this class will check if an argument provides a custom
196  * implementation for formatting.
197  *
198  * <br>In \b C++, again an <b>ALib Boxing</b> interface class is used.
199  * If a given boxed argument provides boxing interface
200  * \ref aworx::lib::strings::boxing::IFormat "IFormat", this interface is invoked and string
201  * \b format_spec is passed for custom processing.
202  *
203  * Information and sample code on how to adopt custom types to support this interface is
204  * found in the documentation of <b>ALib Boxing</b>, chapter
205  * \ref alib_namespace_boxing_strings_iformat "6.4 Interface IFormat".
206  *
207  * Built-in class
208  * \ref aworx::lib::time::Ticks "Ticks" supports custom formatting with boxing interface
209  * \ref aworx::lib::time::IFormat_TTicks "IFormat_TTicks"
210  *
211  *
212  * <p>
213  * - <b>Conversions:</b><br>
214  * In the \b Python placeholder syntax specification:
215  *
216  * "{" [field_name] ["!" conversion] [":" format_spec] "}"
217  *
218  * symbol \c '!' if used prior to the colon <c>':'</c> defines
219  * what is called the <b>conversion</b>. With \b Python, three options are given:
220  * \c '!s' which calls \c str() on the value, \c '!r' which calls \c repr() and \c '!a' which
221  * calls \c ascii(). This is of-course not applicable to this formatter. As a replacement,
222  * this class extends the original specification of that conversion using \c '!'.
223  * The following provides a list of conversions supported. The names given can be abbreviated
224  * at any point and ignore letter case, e.g. \c !Upper can be \c !UP or just \c !u.
225  * In addition, multiple conversions can be given by concatenating them, each repeating
226  * character \c '!'.<br>
227  * The conversions supported are:
228  *
229  * - <b>!Upper</b><br>
230  * Converts the contents of the field to upper case.
231  * - <b>!Lower</b><br>
232  * Converts the contents of the field to lower case.
233  * - <b>!Quote</b><br>
234  * Puts quote characters \c '"' around the field.
235  * Note that these characters are not respecting any optional given field width but instead
236  * are added to such.<br>
237  * An alias name for \!Quote is given with \b !Str. As the alias can be abbreviated to \b !s,
238  * this provides compatibility with the \b Python specification.
239  * - <b>!ESC[<|>]</b><br>
240  * In its default behavior r if \c '<' is specified, certain characters are converted escape sequences.
241  * If \c '>' is given, escape sequences are converted to their (ascii) value. See
242  * \ref aworx::lib::strings::Format::Escape "Format::Escape" for details about the conversion
243  * that is performed.<br>
244  * An alias name for \b !ESC< is given with \b !a which provides compatibility
245  * with the \b Python specification.
246  * - <b>!Tab[Cc][NNN]</b><br>
247  * Inserts fill characters to extend the length of the string to be a multiple of a tab width.
248  * By default the fill character is space <c>' '</c>. It can be changed with optional character
249  * 'C' plus the character wanted. The tab width defaults to \c 8. It can be changed by adding
250  * an unsigned decimal number.
251  * Converts the contents of the field to lower case.
252  * - <b>!ATab[[Cc][NNN]|Reset]</b><br>
253  * Inserts an "automatic tab". These are tabulator positions that are stored internally
254  * and are automatically extend their tab-position in the moment the actual contents exceeds the
255  * currently stored tab-position. An arbitrary amount of auto tab stops are maintained by
256  * the formatter. Which each new invocation of
257  * \ref aworx::lib::strings::Formatter::FormatList "Formatter::FormatList" the
258  * first auto tab stop is chosen and with each use of \c !ATab the next tab stop is used.<br>
259  * By default the fill character is space <c>' '</c>. It can be changed with optional character
260  * 'C' plus the character wanted. The optional number provided gives the growth value by which
261  * the tab will grow if its position is exceeded. This value defaults to \c 3.
262  * The positions currently stored with the formatter can be reset with providing argument
263  * \c Reset. Alternatively to this, outside of a formatting process, the tab stops can be reset
264  * by invoking method
265  * \ref aworx::lib::strings::AutoSizes::Reset "Reset" on field #TabSizes.
266  **************************************************************************************************/
