49 /** ************************************************************************************************
50  * ## Introduction ##
51  * This is a class used as a throwable object with C++ keyword \c throw.
52  *
53  * Often in C++ software, while the stack is unwinded due to a \c throw statement, the original
54  * exception gets replaced (by user code) with a more meaningful one.
55  * For example, if a resource string can not be loaded, a type of "resource exception" may be
56  * thrown. Then, the caller of this method has different information, i.e. what kind of resource
57  * string could not be loaded.
58  *
59  * Instead of such replacement, this class allows to add new information to a list of entries.
60  * Precisely, the information passed with the constructor call of the class, simply becomes the
61  * first entry in the list. This way, all information along the "call stack" is preserved
62  * which allows
63  * - Exception handling code to inspect the entries and perform different actions depending
64  * on the original cause.
65  * - User output that gives more meaningful information about what happened. (Of-course, this is
66  * especially non-productive debug and test runs.)
67  *
68  \~Comment ####################################################################################### \~
69  * ## Custom %Enum Types As %Exception Codes ##
70  * This class is not designed to be a base class for derived exception types. This is because
71  * as explained above, objects of this class should not be replaced, but instead filled with entries
72  * while the stack is unwinded.
73  *
74  * With other implementations of this approach to exceptions, usually an important problem is
75  * raised: What can be used as an error code and how can error codes be kept "non-overlapping"
76  * between different code units and even third party libraries? This class overcomes this
77  * problem by leveraging several \alib concepts:
78  * - Class \alib{lang,Enum} which allows type-safe storage of enumeration values of arbitrary
79  * types (including C++ 11 scoped enums). To reach this, that class leverages
80  * - \ref aworx::lib::boxing "ALib Boxing". This way, runtime-type information about
81  * stored error codes of arbitrary enum elements can be received.
82  * - \alib{lang,T_EnumMetaDataDecl,ALib Enum Meta Data} to equip elements of arbitrary enum
83  * types with static meta information. This, besides other things is used to define
84  * the information text of enum entries.
85  * - \alib{lang,Resources,ALib Resource Data}, which allows to change the information text of
86  * exceptions (e.g. translate to different languages or to tailor descriptions of exceptions
87  * defined in different code libraries to the appropriate wording of the software using such
88  * library).
89  *
90  * Finally, to convert an \b %Exceptions, including its list of entries into user readable
91  * form:
92  * - Exception entries allow to store arbitrary arguments using
93  * \alib{boxing,Boxes, lists of ALib Boxes}, which then can be nicely formatted using
94  * - \alib{strings::format,Formatter,ALib Formatters}.
95  *
96  \~Comment ####################################################################################### \~
97  * ## Preparing A Custom %Enum Type ##
98  * To allow a custom (scoped) enumeration type to be used as an exception error code,
99  * macro\ref ALIB_LANG_EXCEPTIONS is to be used with the type. The macro
100  * does all that is needed:
101  * - \p TEnum is equipped with \alib{lang,T_EnumMetaDataDecl,ALib Enum Meta Data}. The meta
102  * data type is fixed to \alib{lang,Exception::TTuple}.
103  * - The type is \alib{lang,T_Resourced,declared to be loaded} from library resources.
104  * - A custom specialization of \alib{lang,EnumMetaData::CheckLoad} is provided that
105  * reads the meta data tuple table from the resources object of the according \alib{lang,Library}.
106  *
107  * \note
108  * As a consequence, no other meta information must (can) be declared with type \p TEnum
109  * when this macro is used. In cases when more/other meta data is needed for an enum type,
110  * the only way is to create a "mirror" enum class with the same underlying integer types and
111  * probably provide a custom assign operator or a conversion function.
112  * It is up to the user of \alib to solve such requirement externally.
113  *
114  *
115  \~Comment ####################################################################################### \~
116  * ## Throwing Class %Exception ##
117  * With one or more enum types prepared, class Exception can now be created an thrown.
118  * The constructor (as well as method \alib{lang,Exception::Add}) expects caller source
119  * information. Usually those are provided using macro
120  * \ref ALIB_SRCPOS_REL_EMPTY. If the source information should still be
121  * available in release compilations, then alternatively macro \ref ALIB_SRCPOS
122  * may be used, which does not \e "prune" them.
123  *
124  * Then the enumeration element of the prepared enum class type is provided.
125  * Finally, a list of optional parameters can be given. The optional parameters can be used
126  * in two ways:
127  * - Catch handlers might investigate into them to receive detailed information to decide
128  * how to act on an exception.
129  * - Catch handlers can use method #Describe (alternatively leveraging class
130  * \alib{strings::format,SimpleText} or using
131  * \ref aworx::lox::LogTools::Exception "ALox LogTools" to nicely format the output).
132  *
133  * Here, a conflict may arise: Method #Describe passes all optional parameters
134  * to a \alib{strings::format,Formatter} along with a format string loaded from
135  * resources. If now a catch handler needs some information that should not be passed to a
136  * formatter, the way to cope with that is:
137  * - Add the non-formattable arguments as the first one in the list.
138  * - Add a corresponding amount of suppression placeholders <c>{!X}</c> to the start of the format
139  * string.
140  *
141  * This way, the argument will not be displayed and is easily accessible for the catch handler.
142  *
143  * ## Catching Exceptions, Adding Information, Rethrowing ##
144  *
145  * Due to the fact that \alib exceptions are rather heavy objects, they should be caught by
146  * reference (with ampersand <c>'\&'</c>), like this:
147  *
148  * catch( aworx::Exception& e )
149  * {
150  * ...
151  * }
152  *
153  *
154  * In the case an exception should be rethrown, method #Add may be used to add one or more new
155  * entries to the exception instance. Method #Add has the very same signature as the constructor.
156  *
157  * For the same reasons that exceptions should be caught by reference, rethrowing should be
158  * done without the exception argument. While the following code compiles fine:
159  *
160  * catch( aworx::Exception& e )
161  * {
162  * throw e.Add( ... );
163  * }
164  *
165  * it copies the exception. Therefore the following code scheme is preferred:
166  *
167  * catch( aworx::Exception& e )
168  * {
169  * e.Add( ... );
170  * throw;
171  * }
172  *
173  \~Comment ####################################################################################### \~
174  * ## Lifecycle Of Arguments ##
175  *
176  * The arbitrary arguments added to exception entries are internally stored in \alib{boxing,Boxes}.
177  * As documented, fundamental C++ types are boxed as values, while custom types usually get
178  * boxed as pointers (or references). Now, when the stack is unwinded, local objects will
179  * become unavailable and accessing them is not allowed. This has to be kept in mind when adding
180  * arguments: Some hae to be copied before passed and delete later.
181  *
182  * For all string like types (note: all supported, even 3rd party string classes get boxed as
183  * arrays of characters!), for example the name of a file that could not be opened, this class
184  * internally creates a copy already. Therefore, local strings can just be passed.
185  * The destructor of this class deletes all string copies.
186  *
187  * For custom types, custom code is needed to delete the objects in the catch handler.
188  * In the case that the exception is not caught, the occurring memory leak obviously is the smaller
189  * problem.
190  *
191  * To avoid double deletion of objects, the copy and move constructor of the class have been
192  * implemented. They both clear the entry vector of the source object. Hence, Exceptions can not
193  * by "duplicated" by assigning one to another.
194  *
195  \~Comment ####################################################################################### \~
196  * ## Using Negative %Exception Codes ##
197  *
198  * A final feature of class exception is implemented with method \alib{lang,Exception::Code}.
199  * This method usually returns the \alib{lang,Enum} object added to the exception most recently.
200  * But this is only half the truth: In fact, the method returns the last object in the list of
201  * entries that has a positive enumeration value. This way, entries that are purely considered
202  * "additional information" - in contrast to entries that somehow change the meaning of an
203  * exception - may be declared using negative enumeration code. This way, the final
204  * catch handler does not need to switch over just all exception codes that can be thrown, but
205  * just over those that are considered to bring a different meaning to an exception.
206  *
207  \~Comment ####################################################################################### \~
208  * ## Sample Code ##
209  * A well documented sample of the three simple steps, namely:
210  * 1. Define an enum class that lists the exceptions
211  * 2. Set defaults for the externalized resource strings for each exception enum element
212  * 3. Start throwing!
213  *
214  * that are needed to use \alib exceptions is given
215  * \ref cpp_alib_lang_T_EnumMetaDataSpecification_exception_sample "here".
216  *
217  * ## See Also ##
218  * - To generate formatted descriptions of exception entries, method
219  * \alib{strings::format,SimpleText::AddException} may be used.
220  * - With the use of \alox, static tool method \alox{LogTools::Exception} allows to log
221  * exceptions with one line of code.
222  *
223  * # Reference Documentation #
224  **************************************************************************************************/
