21 /** ************************************************************************************************
22  * ## Introduction ##
23  *
24  * This class is used by \alib to manage the library bootstrap and termination, externalized
25  * resources, configuration data and such.
26  *
27  * As \alib itself is modularized, almost every \alib namespace is designed to be used and
28  * distributed solely or with just a smaller foundation part of \alib. Therefore, each \alib module
29  * or namespace uses its own instance of this class.<br>
30  * Of-course, software using \alib, may use this class in the same way as <b>%ALib Modules</b>,
31  * \alox and other A-Worx software does.
32  *
33  *
34  *
35  * \~Comment #################################################################################### \~
36  * ## %Library Hierarchies ##
37  *
38  * Software libraries generally depend on other libraries. Such dependencies on a larger scale
39  * should be non-circular, but on a lower scale this is often not avoidable. As \alib, with the use
40  * of this class treats its namespaces as libraries, there are many circular samples found like:
41  * \ref aworx::lib::config relies on \ref aworx::lib::strings while parts of the string utility
42  * and formatter classes in turn rely on \ref aworx::lib::config. For modularized distribution,
43  * such dependencies often get removed using \ref GrpALibCodeSelectorSymbols and cutting some
44  * functionality that only exists in a more complete or full distribution.
45  *
46  * Now, the approach that \alib takes to bootstrap, share resources and configuration variables
47  * is as follows:
48  * - Each module/namespace derives a library class from this type. The name of the class
49  * corresponds to the namespace name, but written in capital letters.
50  * - A static singleton instance of this library class is placed in namespace
51  * \ref aworx::lib, named the same as the class itself but in capital letters.
52  * - With the two rules above, the library singletons are:
53  * - \ref aworx::lib::threads::Thread \ref aworx::lib::THREADS
54  * - \ref aworx::lib::system::System \ref aworx::lib::SYSTEM
55  * - \ref aworx::lib::strings::Strings \ref aworx::lib::STRINGS
56  * - \ref aworx::lib::boxing::Boxing \ref aworx::lib::BOXING
57  * - \ref aworx::lib::config::Config \ref aworx::lib::CONFIG
58  * - etc... (See class inheritance diagram above for a complete list and namespace documentation
59  * \ref aworx::lib for the singletons.)
60  * - In addition to this, there is one central \alib library found with static field:
61  * - \ref aworx::lib::ALib \ref aworx::lib::ALib
62  * - In the constructor, dependency libraries are added to field #dependencies as needed and
63  * by using \ref GrpALibCodeSelectorSymbols with modularized \alib distributions.
64  *
65  *
66  * \~Comment #################################################################################### \~
67  * ## %Resources and %Configuration ##
68  *
69  * This class has pointer fields #Config and #Res. Classes of a library will use these
70  * instances by addressing them through the singleton of their library found in namespace
71  * \ref aworx::lib (or through wherever a custom library places its singleton).
72  *
73  * A library has to be aware that these objects may be shared with a lot of other libraries.
74  * Therefore, both classes provide a \ref aworx::lib::threads::ThreadLock "thread safe" access
75  * when used. Furthermore, a consequence is that clear and understandable resource and variable
76  * categories and names are to be used, to avoid conflicting entries.
77  *
78  * Among other things, the next sections explain how a user may control which libraries share
79  * resources and/or configuration singletons, respectively how to initialize libraries independently
80  * for not sharing such objects.
81  *
82  *
83  * \~Comment #################################################################################### \~
84  * ## Bootstrapping ##
85  *
86  * The C++ language specification misses rules and clarity about how static data - whose types might
87  * execute constructor code - are initialized, especially in which order. Therefore, in this
88  * language it is no option to create "self-initializing" libraries.
89  * Bootstrapping becomes even more complicated with the use of resources and configuration variables
90  * that both may be equipped with custom data settings or even plug-ins that load data from custom
91  * places.<br>
92  * This class therefore tries to streamline and simplify bootstrapping.
93  *
94  * Implementations of this class need to implement two protected, \b abstract methods:
95  * - #init and
96  * - #terminationCleanUp.
97  *
98  * While the latter is invoked only once, the initialization is done in phases defined with
99  * #Phases. There are three phases defined and the implementations of #init need to
100  * switch upon them:
101  * - \alib{lang,Library::Phases::resourceset,Phases::resourceset}<br>
102  * In this phase is is assured that all static objects of C++ got initialized (because
103  * bootstrapping is called explicitly from a code executed within the C++ <c>main()</c>
104  * function).<br>
105  * A library that uses resources needs to add default values of such resources in this phase
106  * to field #Res. This is done either "hard coded" or using a third party resource
107  * loader.
108  * - \alib{lang,Library::Phases::resourceload,Phases::resourceload}<br>
109  * In this phase, resources that are <b>not lazily loadable</b> need to be loaded here explicitly.
110  * Samples of lazily loadable resources are:
111  * - \alib{strings,ResourceString}
112  * - Resourced \alib{lang,T_EnumMetaDataDecl,ALib Enum Meta Data} (when using the built-in
113  * implementation)
114  * - Resourced \alib \ref ALIB_CONFIG_VARIABLES "configuration variable declarations".
115  * - Resourced \alib \ref ALIB_LANG_EXCEPTIONS "exceptions".
116  * - \alib{lang,Library::Phases::final,Phases::final}<br>
117  * In this phase, final initialization steps may be performed. Especially, configuration
118  * variables are (only) available now.
119  * As a sample, in this phase, class \ref aworx::lib::ALib sets the processes' \e locale depending
120  * on configuration variable \b ALIB_LOCALE.<br>
121  *
122  * Now, to initialize a library, one of the overloaded #Init methods is to be called by the using
123  * software on the "highest" level of libraries. This means init has to be invoked on the minimum
124  * subset of libraries that together (recursively) through their dependencies just do initialize all
125  * others.
126  * It is no problem if several libraries share the same dependencies, or circular dependencies
127  * exist. Each library is initialized only once.
128  *
129  * The #Init method will create instances of classes
130  * \alib{lang,Resources} and
131  * \alib{config,Configuration} and recursively pass these to underlying libraries. Furthermore,
132  * the implementations of abstract #init methods will be recursively called (with a depth first
133  * algorithm).
134  *
135  * All overloaded #Init methods have parameter \p level, which defaults to
136  * \alib{lang,Library::InitLevels::All,InitLevels::All}. In a next section it is explained when
137  * to specify this parameter to a different value.
138  *
139  *
140  * \~Comment #################################################################################### \~
141  * ## Bootstrapping With Custom %Resources and/or %Configuration Plug-ins##
142  *
143  * As documented with classes \alib{lang,Resources} and \alib{config,Configuration}, both classes
144  * allow to be equipped with custom mechanics for loading external data. To do so, the bootstrapping
145  * of \alib libraries need to be explicitly invoked for separated phases, by setting parameter
146  * \p level of the #Init methods as follows:
147  * - <b>To create object #Res and fill it with default values:</b><br>
148  * - #Init needs to be called with parameter \p level set to
149  * \alib{lang,Library::InitLevels::PrepareResources,InitLevels::PrepareResources}.
150  * - After that, object #Res may be equipped with custom resource loading mechanics.
151  * - Now, #Init needs to be called again, e.g. with default value of \p level,
152  * \alib{lang,Library::InitLevels::All,InitLevels::All}
153  *
154  * - <b>To add custom configuration plug-ins:</b><br>
155  * - #Init needs to be called with parameter \p phases set to
156  * \alib{lang,Library::InitLevels::PrepareConfig,InitLevels::PrepareConfig}
157  * - After that, object #Config may be equipped with a custom plug-in, e.g. of type
158  * \alib{config,IniFile}.
159  * - Now, #Init needs to be called again with \p phases set to
160  * \alib{lang,Library::InitLevels::All,InitLevels::All}.
161  *
162  * Of-course, both approaches can be mixed. In this case, the following order has to be adhered:
163  * - Initialize with \alib{lang,Library::InitLevels::PrepareResources,InitLevels::PrepareResources}
164  * - Customize #Res
165  * - Initialize with \alib{lang,Library::InitLevels::PrepareConfig,InitLevels::PrepareConfig}
166  * - Customize #Config
167  * - Initialize with \alib{lang,Library::InitLevels::All,InitLevels::All}
168  *
169  *
170  * \~Comment #################################################################################### \~
171  * ## Using Different %Resources and Configuration Instances ##
172  *
173  * As explained above, the variants of #Init create corresponding class instances for fields
174  * #Res and #Config and pass them recursively down to dependent libraries. Now, if
175  * different, dedicated instances of these objects should be used with different libraries,
176  * the the simple approach is to call the #Init methods explicitly. Of-course, "lower level"
177  * libraries need to be initialized first, higher level next, as otherwise the higher level
178  * libraries would already initialize the lower ones.
179  *
180  * The goal of doing so is separate resources and configuration variables of different libraries
181  * in different external configuration files, databases or whatsoever.
182  *
183  * To get a better understanding lets look at a sample. Requirements should be:
184  * - An application <b>"MyApp"</b> uses \alox logging library, which depends on \alib, as documented with \alox
185  * library class \ref #aworx::lox::ALox
186  * - The application uses \alib library mechanics (hence a singleton of a custom type derived from this class!)
187  * - The application uses class \alib{config,IniFile} to load custom \alib{config,Variable} objects.
188  * - The application wants to store \alib and \alox variables in separated files, just to
189  * remove "clutter" from its main configuration file and document the use of the \alib/\alox
190  * specific variables separated from the main configuration files' documentation.
191  *
192  * What has to be done is:
193  * - Initialize \alox with level
194  * \alib{lang,Library::InitLevels::PrepareConfig,InitLevels::PrepareConfig}
195  * - Add a custom configuration plug-in (e.g \alib{config,IniFile}) to singleton
196  * #aworx::lox::ALOX.
197  * - Initialize \alox with level \alib{lang,Library::InitLevels::All,InitLevels::All}
198  *
199  * Now \alox and all underlying \alib modules are initialized and all share the same configuration
200  * and resource objects.
201  *
202  * - Then do the same with the application specific library singleton and provide a different
203  * custom plug-in to it's configuration.
204  *
205  * With this approach, both objects resources and configuration are separated.
206  * If just the configuration should be separated, the steps to be done are:
207  * - Make library singleton \ref #aworx::lox::ALOX a dependency library of <b>"MyApp"</b>.
208  * (This is only an optional step in the previous to do list.)
209  * - Initialize <b>"MyApp"</b> with level
210  * \alib{lang,Library::InitLevels::PrepareResources,InitLevels::PrepareResources}.<br>
211  * (Now all libraries share the same resources!)
212  * - Initialize \alox with level
213  * \alib{lang,Library::InitLevels::PrepareConfig,InitLevels::PrepareConfig}
214  * - Add custom configuration plug-in (e.g \alib{config,IniFile}) to singleton
215  * #aworx::lox::ALOX.
216  * - Initialize \alox with level \alib{lang,Library::InitLevels::All,InitLevels::All}.<br>
217  * (Now \alox is fully initialized with \alox and \alib sharing their configuration.)
218  *
219  * - Initialize <b>"MyApp"</b> with level
220  * \alib{lang,Library::InitLevels::PrepareConfig,InitLevels::PrepareConfig}
221  * - Add custom configuration plug-in (e.g \alib{config,IniFile}) to <b>"MyApp"</b>
222  * - Initialize <b>"MyApp"</b> with level \alib{lang,Library::InitLevels::All,InitLevels::All}
223  *
224  *
225  * \~Comment #################################################################################### \~
226  * ## Custom Methods ##
227  *
228  * A library might decide to add custom library methods, mostly static functionality to the
229  * the library specific derived class. In case of static methods, those could alternatively be
230  * namespace functions instead. It is up to the library designer what to incorporate. In general,
231  * the use of a static method might be more readable in code than just a namespace function.
232  * Furthermore, custom library types of \alib itself are \ref aworx "mirrored in namespace aworx"
233  * and thus can be used without their full namespace prefix when just statement
234  *
235  * using namespace aworx;
236  *
237  * is placed in a compilation unit, which is always advisable.
238  *
239  * \~Comment #################################################################################### \~

572  *
573  * If used, the following steps have to be performed:
574  * - By the library that offers the feature:
575  * - Expose a preprocessor macro that collects the bits at compile time.
576  * As a sample see macros
577  * \ref ALIB_COMPILATION_FLAGS or
578  * \ref ALOX_COMPILATION_FLAGS.
579  * - The constructor of the library needs to be in a compilation unit of the library
580  * (not inline) and needs to pass the macro exposed as a parameter to this class'
581  * constructor.
582  * - The constructor of the library needs to fill field #CompilationFlagMeanings.
583  *
584  * - The user of the library now invokes this method in bootstrap, hence preferably at
585  * the beginning of method method #init with \alib{lang::Library,Phases::resourceset}.
586  *
587  * In case they are different in a way that \b %ALib gets incompatible (e.g. different
588  * class sizes, which results in errors which are confusing the debugger), the flags are
589  * written to \e cout for comparison and \c false is returned.
590  *
591  * @param compilationFlags The flags as defined in invoking compilation unit.
592  * Defaults to \c ALIB_COMPILATION_FLAGS.
