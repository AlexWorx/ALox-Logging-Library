aworx Namespace Reference

Namespace Description

This is the outer C++ namespace for all classes published by A-Worx GmbH, Germany.

Note
While the Java language strongly proposes to use the complete 'reverse domain' name and therefore Java code published by A-Worx GmbH resides in namespace com.aworx, for C# we replaced the prefix com. with cs., which leads to cs.aworx.
This was needed for having classes with the same names existing in C++, C# and Java not collide within the Doxygen documentation system.

As far as it is planned today, there is nothing inside this namespace but other namespaces and - and this is true only for the C++ side of things - type definitions used as shortcuts.

Type Shortcuts in the C++ Namespace "aworx"

ALib and other libraries of A-Worx, "mirror" most classes into namespace aworx with using statements. For example, by having 

using  String =  aworx::lib::strings::String;

stated within namespace aworx in the header file of class String, this class is made available with just adding 

using  aworx;

to a source code. This is convenient when ALib is used, and also convenient for the classes of ALib itself and for 'sibling' libraries that reside underneath namespace aworx, for example logging library ALox with namespace aworx::lox. Such classes may refer to all mirrored classes in their (external!) header files without providing any namespace, because in C++, code that is located inside a namespace 'sees' all types defined in parent namespaces.
Therefore, header files become clear and simple without explicit namespace names.

In contrast to this, C++ sources that are "using" a sub-namespace do not automatically 'see' classes in parent namespaces. This is an important difference (and great design of the C++ language) that we are leveraging with this concept.

Unfortunately, with template classes this mechanism has a limit. It is not allowed to use the same type name for the mirrored classes. Therefore, for example class PreallocatedString<int TCapacity> gets mirrored to PAString as follows: 

template<int TCapacity>
using PAString= aworx::lib::strings::PreallocatedString<TCapacity>;

To summarize, the effect of mirroring the main interface classes is as follows:

  • Inside header files of sub-namespaces (and their sub-namespaces) of aworx, e.g. aworx::lib and aworx::lox, these names are visible and used.
  • Inside external header files, they are not visible but might be quite shortly accessed using namespace prefix aworx::.
  • Inside external cpp files, if no collision with other libraries or local classes exist, all types of all A-Worx libraries used, can be made visible with one single using statement for namespace aworx. Should collisions occur, then still selected classes of A-Worx libraries that do not collide can be made visible by choosing the complete namespace name for the using statement.
  • The mirrored class names appear only in this namespace aworx, if the corresponding header file of the original class is included in a compilation unit.

Domain-Specific Constants and Variables

Similar to the concept of mirroring important interface classes into this namespace aworx (which is described in the previous paragraph), it is also agreed to place important global constants here. The impact and benefits are quite the same.
As a sample, see aworx::PathSeparator which appears in this namespace as soon as header "alib/system/directory.hpp" is included.
