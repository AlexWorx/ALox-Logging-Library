60  * This class primarily is used via its public static singleton instance
61  * \ref aworx::lib::config::Configuration::Default "Configuration::Default".
62  * It holds a list of objects of type
63  * \ref aworx::lib::config::ConfigurationPlugin "ConfigurationPlugin",

329  /** ****************************************************************************************
330  * Writes the variable to the configuration.
331  * In general, this is done by asking each of our installed plug-ins - in the order of
332  * their priority - to do so.
333  * As soon as the first plug-in returns \c true, the variable is considered written and no
334  * other plug-in is asked.
335  * This way, the variable is stored by the plug-in with the highest priority.
336  *
337  * The maximum priority to start the loop with depending on field
338  * \ref aworx::lib::config::Variable::Priority "Variable::Priority"
339  * of the given \p variable. The rules are as follows:
340  * - If the value is \c -1, which is the default value of new variables or ones that
341  * were freshly defined, then prior to storing the value, the variable is (tried to be)
342  * loaded first (without actually overwriting the values).
343  * After that, one of the next two rules apply.
344  * - If the value is \c 0, which indicates that a previous load operation failed, the
345  * loop starts with #PrioDefault (and usually ends there, as standard configuration sets
346  * do not have plug-ins with lower priorities installed). In other words, newly created
347  * variables are stored in the in-memory plug-in #DefaultValues. This way, they are
348  * not written to external configuration files, unless the application explicitly moves
349  * such new default values to dedicated other plug-ins (e.g. on termination).
350  * - If the value is greater than \c 0, the value is used as the start of the loop.
351  * This way, an already defined variable will be stored in the same plug-in as it was
352  * found (or one with a lower priority, if that plug-in does not provide writing
353  * capabilities).
354  *
355  * Consequently, as field
356  * \ref aworx::lib::config::Variable::Priority "Variable::Priority" is public, the behavior
357  * can be manipulated, by setting the field explicitly prior to invoking this method.
358  * Some frequent use cases shall be named here:
359  * - Setting the field to <c>Configuration::PrioProtected</c> allows to store the
360  * variable in plug-in #ProtectedValues. This way, the variable gets \e 'locked' in
361  * respect to external manipulation.
362  * - Setting the field to <c>Configuration::PrioDefault</c> allows to store the
363  * variable in plug-in #DefaultValues. This way, the variable becomes just a default
364  * and does not overwrite other external values.
365  * - Setting the field to a distinct priority value that names a user-specific configuration
366  * (vs. a system-wide configuration).
367  * - Setting the field to <c>Configuration::PrioProtected - 1</c> allows to store the
368  * variable just in the plug-in with highest possible priority, for example
369  * a user specific configuration is preferred to a system wide configuration)
370  * - A variable might be related to a second one. If the priority of the second one is
371  * known, that priority might be set to possibly have both variables stored in the same
372  * plug-in.
373  *
374  * The method returns the priority of the configuration plug-in that the value was written
375  * to as well as storing this value in field
376  * \ref aworx::lib::config::Variable::Priority "Variable::Priority".
377  * If the result is \c 0, the variable was not written. This might only happen if
378  * - either field #DefaultValues was modified (removed or exchanged with a different
379  * plug-in that does not write the value)
380  * - or if field \p Priority of the variable was set below #PrioDefault and greater
381  * than \c 0.
382  * - The detected (!) priority was #PrioProtected. In this case, obviously the application
383  * does not want to allow changes and writing the variable into a different plug-in
384  * has no effect. This way, such variables also do not appear in a users' configuration
385  * in the case that on program termination, new default values are copied there.
386  *
387  * Optional parameter \p externalizedValue allows to provide a string that is parsed
388  * by the storing plug-in to reset the variables' values prior to writing.
389  *
390  * @param variable The variable object.
391  * @param externalizedValue Optional externalized value string. If given, the variable
392  * is set prior to writing.
393  *
394  * @returns The priority of the configuration plug-in that the value was written to.
395  ******************************************************************************************/
396  ALIB_API
397  int Store( Variable& variable, const String& externalizedValue= nullptr );
