ALox Symbols to Select Code Fragments

Detailed Description

Symbols (C++ preprocessor macros) listed here, are mostly generated out of ALox Compilation Symbols. In this case, their name corresponds to these symbols, excluding the suffixes "_ON" and "_OFF".

These symbols must not be passed to the compiler and must also not be set manually from within code entities that use ALox libraries. They can be used in code entities that use ALox to select code depending on the same settings of the ALox library compilation. Selection takes place, using preprocessor directives:

#if defined(ALOX_XYZ_SELECTOR)
// your selective code
#endif

ALox inherits further compiler symbols from underlying library ALIB. These are documented in: ALib Symbols To Select Code Fragments.

(Besides ALox specific symbols of that type, ALox internally uses also platform specific symbols to select code, e.g. __GLIBCXX__, _MSC_VER, etc.).

Macro Definition Documentation

◆ ALOX_DBG_LOG

#define ALOX_DBG_LOG

If defined as true, debug Log Statements are enabled. Hence, users of ALox can conditionally compile special debug logging code that normally belongs to corresponding debug logging statements. This is useful for example, to conditionally compile code that calculates and gathers information to do some more complex log output.

See ALOX_DBG_LOG_ON and ALOX_DBG_LOG_OFF for information about how to manipulate this symbol.

◆ ALOX_DBG_LOG_CI

#define ALOX_DBG_LOG_CI

If defined, scope information is passed to ALox on invocation of debug Log Statements. Hence, users of ALox can conditionally compile special debug logging code that normally belongs to corresponding debug logging statements based on this symbol. A sample would be the definition of different log line meta information formats, depending on the availability of scope information.

Note
The ALox Scope Domain mechanism as well as the ALox features Lox.Once and Log Data rely on caller information. These mechanisms are therefore not available to debug logging unless this symbol is set.

See ALOX_DBG_LOG_CI_ON and ALOX_DBG_LOG_CI_OFF for information about how to manipulate this symbol.

◆ ALOX_REL_LOG

#define ALOX_REL_LOG

If defined, release Log Statements are enabled. Hence, users of ALox can conditionally compile special debug logging code that normally belongs to corresponding debug logging statements. This is useful for example, to conditionally compile code that calculates and gathers information to do some more complex log output.

See ALOX_REL_LOG_ON and ALOX_REL_LOG_OFF for information about how to manipulate this symbol.

◆ ALOX_REL_LOG_CI

#define ALOX_REL_LOG_CI

If defined, scope information is passed to ALox on invocation of release log statements when using the ALox Macros For Release Logging.
Hence, users of ALox can conditionally compile special debug logging code that normally belongs to corresponding debug logging statements based on this symbol. A sample would be the definition of different log line meta-information formats, depending on the availability of scope information.

Note
The ALox Scope Domain mechanism as well as the ALox features Lox.Once and Log Data rely on caller information. These mechanisms are therefore not available to debug logging unless this symbol is set.

See ALOX_REL_LOG_CI_ON and ALOX_REL_LOG_CI_OFF for information about how to manipulate this symbol.
