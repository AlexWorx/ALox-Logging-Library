166 /**
167  * @defgroup GrpMacrosDebugLog ALox Macros For Debug Logging
168  *
169  * The macros listed here, are provided to place debug <em>Log Statements</em> within source code
170  * using \b %ALox.
171  * Besides that, macros controlling and setting preferences for \b %ALox exists.<p>
172  *
173  * The exclusive use of these macros should be sufficient to support most of common debug logging
174  * statements with \b %ALox. Should some functionality be not available easily with using the macros,
175  * of-course, the normal C++ API of \b %ALox can be used in parallel to using the macros.
176  * For proper pruning of code that is using the C++ API, such code has to be enclosed by<br>
177  \verbatim
178  #if ALOX_DBG_LOG
179  ...
180  #endif
181  \endverbatim
182  * lines, or embedded in macro \ref Log_Prune.
183  * \note To be clear: When using the macros, the <em>\#if / \#endif</em> statements are \e not needed!
184  *
185  * All macro names are prefixed "Log_". This implies that they are macros to implement *debug*
186  * logging. In contrast to this, a set of similar macros exists for release logging
187  * (see \ref GrpMacrosReleaseLog). Those are
188  * prefixed "Lox_". (The choice of the prefixes *Log_* and *Lox* provide maximum source code
189  * compatibility of <b>%ALox for C++</b> log lines in comparison to ALox for C# and ALox for Java.<p>
190  *
191  * Most macros make use of the macro \ref LOG_LOX, which references a singleton
192  * object of class \ref aworx::lox::Lox "Lox" that is used for all debug logging.
193  * This singleton concept covers most
194  * uses cases for debug logging. If more flexibility is wanted, then either macro
195  * \ref LOG_LOX might be changed for different compilation units or the <b>%ALox for C++ API</b>
196  * might be used instead of the macros listed here.
197  *
198  * \note The <em>Scope Domain</em> mechanism of \b %ALox, as well as other \b %ALox functionality
199  * which relies on \e Scopes use the information provided by \b %ALib macro \ref ALIB_SRCPOS.
200  * Changing this macro, might cause these mechanism to fail.
201  * @{
202  */

308  * @defgroup GrpMacrosReleaseLog ALox Macros For Release Logging
309  *
310  * The macros listed here, are provided to place release <em>Log Statements</em> within source code using
311  * \b %ALox. Besides that, macros controlling and setting preferences for ALox exists.<p>
312  *
313  * The exclusive use of these macros should be sufficient to support most of common release logging
314  * statements with \b %ALox. Should some functionality be not available easily using the macros,
315  * of-course, the normal C++ API of \b %ALox can be used in parallel to using the macros.
316  * Code that is using the C++ API might be enclosed by<br>
317  \verbatim
318  #if ALOX_REL_LOG
319  ...
320  #endif
321  \endverbatim
322  * lines, to remove them when compiling a release version of the software unit with pruned
323  * release log macros. Alternatively, such code might be embedded in macro \ref Lox_Prune.
324  * (Such compilation would be made by passing \ref ALOX_REL_LOG_OFF to the
325  * compiler and could be useful in certain situations.) <p>
326  *
327  * Prior to using the macros, each code entity has to set the preprocessor macro \ref LOX_LOX
328  * This can be done in a general header file of the software,
329  * (e.g. the same that exposes the release-Lox object to that source), or, in more complex
330  * scenarios with more than one release-Lox object, at any other appropriate source location.
331  *
332  * All macro names are prefixed with the term <b>Lox_</b>. This implies that they are macros to
333  * implement <em>release logging</em>. In contrast to this, a set of similar macros exists for
334  * <em>debug logging</em> (see \ref GrpMacrosDebugLog). Those are
335  * prefixed <b>Log_</b>.
336  * \note The choice of the prefixes <b>Log_</b> and <b>Lox_</b> was made to provide maximum source
337  * code compatibility of <b>%ALox for C++</b> log lines in comparison to other ALox implementations
338  * in other programming languages, which provide a class Log as a kind of 'mirror' class of
339  * class Lox. Invocations to class Log get pruned in these languages. <p>
340  *
341  * Most macros make use of the macro \ref LOX_LOX, which references a singleton
342  * object of class \ref aworx::lox::Lox "Lox" that is used for all release logging.
343  * This singleton concept covers most
344  * uses cases for release logging. If more flexibility is wanted, then either the macro
345  * \ref LOX_LOX might be changed for different compilation units or the <b>%ALox C++ API</b>
346  * might be used instead of the macros listed here.
347  * @{
348  */
