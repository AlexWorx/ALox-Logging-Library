101 /** ************************************************************************************************
102  * \note
103  * This class is extracted and exposed as an own <b>ALib Module</b>, found here:
104  * [ALib Singleton](https://github.com/AlexWorx/ALib-Singleton).
105  *
106  * <b>Background Information on Template Singletons and Windows OS DLLs:</b><br>
107  * This template class implements the singleton design pattern using templates. In principle, this
108  * is a very easy and straightforward thing and extensively discussed in the internet and
109  * programming literature. However, in combination with using DLLs on Windows OS, things get a
110  * little more complicated. A Windows OS DLL provides its own data segment and static function
111  * variables defined in template classes (in header files!) are duplicated within DLLs and
112  * the main executable. Now, if a standard implementation of the singleton pattern with templates
113  * is used, the singletons received are "DLL-wide singletons" but not "process-wide singletons".
114  *
115  * Therefore, this implementation uses one static hash map to store each singleton instance of
116  * any type. Then, if a singleton is requested for the first time, this hash map is consulted
117  * for an already existing instance. Only if this fails, the singleton instance is created and
118  * stored in the hash map to be found by other DLLs or the main executable.<br>
119  * The pointer to the singleton created or found is then stored in static field #singleton of this
120  * class. This way, a second request for the singleton (from within the same software entity), will
121  * not have the need to search the map again.
122  * In other words, the data model is as follows: each derived template type will have exactly one
123  * singleton instance, stored in the static hash map which is defined only once (in the \b %ALib
124  * DLL or library). Software entities that have their own data segment (e.g. Windows DLLs) each
125  * have one dedicated pointer to the singleton created by the entity itself or by one of the
126  * other entities.
127  *
128  * <b>Switching Compilation Mode:</b><br>
129  * As said above, the effort done with this implementations is needed only if a software process
130  * uses different data segments for different portions of the code, for example processes which open
131  * DLLs on Windows OS.
132  * As this effort imposes a (small) overhead, the implementation of this class can be switched to
133  * a simple implementation version, which just stores the singletons in a static field of the class.
134  * The implementation is controlled by code selection symbol \ref ALIB_FEAT_SINGLETON_MAPPED.
135  * On Windows OS the symbol defaults to \c true, otherwise to false.<br>
136  * Now, Compiler symbols \ref ALIB_FEAT_SINGLETON_MAPPED_ON and \ref ALIB_FEAT_SINGLETON_MAPPED_OFF
137  * can be used to overwrite this default settings. This could be done for the following reasons:
138  * - \b %ALib is used on Windows OS, but not compiled as a DLL and the code that is using \b %ALib
139  * is restricted to one DLL or the main application exclusively. In this case, symbol
140  * \ref ALIB_FEAT_SINGLETON_MAPPED_OFF may be passed to the compiler to switch compilation
141  * to the simple mode.
142  * - \b %ALib is used on GNU/Linux, but the debug features for listing all singletons created
143  * (described below) should be used. In this case, for the purpose of debugging, symbol
144  * \ref ALIB_FEAT_SINGLETON_MAPPED_ON may be passed to the compiler to switch compilation
145  * to use the hash map, even though this is not needed for that platform.
146  *
147  * <b>Usage:</b><br>
148  * The implementation details described above are transparent in respect to using the class.
149  *
150  * A user type may inherit the singleton feature from this class by being derived from it,
151  * with the template parameter \p TDerivedClass set to name just of the derived class type itself.
152  * Then, static method #GetSingleton will return a singleton of the derived type.
153  *
154  * Here is a sample code deriving from this class:
155  *
156  * \snippet "DOX_ALIB_LANG_SINGLETON.cpp" DOX_ALIB_LANG_SINGLETON_1
157  *
158  * Now, the singleton can be requested as follows:
159  *
160  * \snippet "DOX_ALIB_LANG_SINGLETON.cpp" DOX_ALIB_LANG_SINGLETON_2
161  *
162  * Note that the singleton is \b not automatically the only instance of the
163  * custom class. In this respected, class \b %Singleton is not \e 'strict':
164  *
165  * \snippet "DOX_ALIB_LANG_SINGLETON.cpp" DOX_ALIB_LANG_SINGLETON_3
166  *
167  * To create classes that implement strict singletons, two things have to be done:
168  * 1. Provide a private constructor to the derived class and
169  * 2. make parent class \b %Singleton be a friend of it.
170  *
171  * This is shown in the following sample:
172  * \snippet "DOX_ALIB_LANG_SINGLETON.cpp" DOX_ALIB_LANG_SINGLETON_STRICT_1
173  * \snippet "DOX_ALIB_LANG_SINGLETON.cpp" DOX_ALIB_LANG_SINGLETON_STRICT_2
174  *
175  * Prior to terminating the process, the singletons can be destructed and deleted using namespace function
176  * \ref aworx::lib::lang::DeleteSingletons. See this method's description for more information.
177  * Especially Note that method
178  * \ref aworx::lib::ALIB::TerminationCleanUp "ALIB::TerminationCleanUp" invokes this method already.
179  *
180  *
181  * <b>Restrictions / Penalties:</b><br>
182  *
183  * - The singleton class needs to have a default (parameterless) constructor.
184  * (This might be overcome with a templated get-method that forwards variadic arguments, but
185  * is not implemented here.)
186  * - The singleton class is virtual and hence all derived classes become virtual. This is needed
187  * to be able to have a virtual destructor.
188  * - There is a minimal performance penalty in comparison to more simple implementations of the singleton
189  * design pattern: Each <em>'code entity'</em> (DLL or main executable)
190  * needs to retrieve a singleton which was potentially already created by another code
191  * entity, \b once. In other words, only the first request for the singleton incorporates two
192  * small performance penalties for
193  * 1. setting and releasing a thread lock (mutex)
194  * 2. retrieving the singleton or inserting the singleton if not retrieved.
195  * - The memory penalty is also quite minimal and imposed by the creation of a static hash table,
196  * which holds a pointer to each Singleton.
197  * - Upon exit of the process, programmers might want to explicitly free the hash table to avoid
198  * the detection of memory leaks by metrics tools like [Valgrind](http://valgrind.org/).
199  * (Otherwise this can be omitted, as the memory is cleaned by the OS probably much faster when a
200  * process exits).
201  * - In general, it is advised to think twice before implementing the singleton design pattern.
202  * Often, this is considered bad code design. However, there can be very good reasons for having
203  * singletons in software.
204  * And do not be frustrated if "very wise people" tell you that you must never use a singleton.
205  * Your answer to them is: Each and every created instance of a virtual C++ class contains
206  * a pointer to a singleton, namely the [vtable](https://en.wikipedia.org/wiki/Virtual_method_table)!
207  *
208  * <b>Listing Types in Debug Compilations:</b><br>
209  * As described above, to extend the singleton concept across DLL bounds, a static hash map is
210  * needed. Now, in debug compilations (see compilation symbol #ALIB_DEBUG), this map can be
211  * dumped any time and this way all singletons created at a certain point at runtime can be
212  * listed.
213  * For this, \b %ALib provides namespace function
214  * \ref aworx::lib::debug::GetSingletons "GetSingletons".
215  * This function is using class AString, and hence it is available only with \b %ALib distributions that
216  * includes module <b>ALib String</b>.
217  * Fore example modules
218  * [ALib Singleton](https://github.com/AlexWorx/ALib-Singleton) or
219  * [ALib %Boxing](https://github.com/AlexWorx/ALib-Boxing) are missing this method.
220  * The good news is, that it is quite easy to create an own debug method, just as the
221  * following sample shows:
222  *
223  * \snippet "DOX_ALIB_LANG_SINGLETON.cpp" DOX_ALIB_LANG_SINGLETON_DEBUG_LIST_1
224  * \snippet "DOX_ALIB_LANG_SINGLETON.cpp" DOX_ALIB_LANG_SINGLETON_DEBUG_LIST_2
225  *
226  * @tparam TDerivedClass Template parameter that denotes the name of the class that implements
227  * the singleton.
228  *
229  **************************************************************************************************/
230 template <typename TDerivedClass>
231 class Singleton
232 {
233  protected:
234  /** A pointer to the one and only singleton. */
235  static TDerivedClass* singleton;
236 
237  public:
238  /**
239  * Creates (if not done, yet) and returns the singleton of type \p TDerivedClass.
240  * @return The singleton instance.
241  */
242  inline static TDerivedClass* GetSingleton()
243  {
244  if( !singleton )
245  {
246  #if ALIB_FEAT_SINGLETON_MAPPED
247  Singleton<TDerivedClass>* castedAsSingleton;
248  if( !getSingleton(typeid(TDerivedClass), &castedAsSingleton ) )
249  {
250  singleton= new TDerivedClass();
251  castedAsSingleton= dynamic_cast<Singleton<TDerivedClass>*>( singleton );
252 
253  storeSingleton( typeid(TDerivedClass), castedAsSingleton );
254  }
255  else
256  {
257  singleton= dynamic_cast<TDerivedClass*>( castedAsSingleton );
258  }
259  #else
260  singleton= new TDerivedClass();
261  #endif
262  }
263  return singleton;
264  }
265 
266  /** Virtual destructor. */
267  virtual ~Singleton()
268  {
269  #if ALIB_FEAT_SINGLETON_MAPPED
270  removeSingleton( typeid(TDerivedClass) );
271  #endif
272  }
273
