ALox Compilation Symbols

Detailed Description

Symbols (C++ preprocessor macros) listed here, support adding or removing features to ALox when compiling the library and any code unit that uses ALox. These symbols must be used only within makefiles or IDE project files to be passed to the C++ compiler (normally -D option). In other words: they must not be defined or undefined within the source code that uses the ALox library.

For most of them, a version with a suffix "_ON" and "_OFF" exists. If both are set at the same time, a compiler error is generated by ALox header files. If none of them is passed to the compiler, a default value is assumed.

As a convenience, to get default behavior, no symbol needs to be passed to the compiler.

ALox inherits further compiler symbols from underlying library ALIB. These are documented in: ALib Compilation Symbols.

Note
Some of the compilation symbols are changing the definition of classes and methods which makes binaries incompatible with binaries that do not use the same setting for each of these symbols.
Symbols that have to be kept equal across compilation units are denoted in this documentation.

Macro Definition Documentation

#define ALOX_DBG_LOG_CI_OFF

Disables the generation of scope information using the built-in preprocessor macros like "__FILE__" and "__LINE__" for ALox debug logging statements

Passing this Symbol to the compiler, makes ALox to leave the code selection symbol ALOX_DBG_LOG_CI undefined.
Must not be used in conjunction with ALOX_DBG_LOG_CI_ON.

If neither ALOX_DBG_LOG_CI_ON nor ALOX_DBG_LOG_CIO_OFF are specified, then ALOX_DBG_LOG_CI_ON is the default.

Note
The ALox Scope Domain mechanism as well as the ALox features Lox.Once and Log Data rely on caller information. These mechanisms are therefore not available to debug logging when this symbol is set.
#define ALOX_DBG_LOG_CI_ON

Explicitly enables the generation of scope information using the built-in preprocessor macros like "__FILE__" and "__LINE__" for ALox debug logging statements

Passing this Symbol to the compiler, makes ALox define the code selection symbol ALOX_DBG_LOG_CI. Must not be used in conjunction with ALOX_DBG_LOG_CI_OFF.

If neither ALOX_DBG_LOG_CI_ON nor ALOX_DBG_LOG_CIO_OFF are specified, ALox will enable caller source information for debug logging. In other words, ALOX_DBG_LOG_CI_ON is the default and passing the symbol to the compiler is not mandatory.

Note
The ALox Scope Domain mechanism as well as the ALox features Lox.Once and Log Data rely on caller information. These mechanisms are therefore not available to debug logging unless this symbol is set.
#define ALOX_DBG_LOG_OFF

Explicitly disables debug logging by leaving the code selection symbol ALOX_DBG_LOG undefined. Any debug Log Statements that are embedded in code by using the ALox Macros For Debug Logging, will get pruned.

Must not be used in conjunction with ALOX_DBG_LOG_ON.

If neither ALOX_DBG_LOG_ON nor ALOX_DBG_LOG_OFF are specified, ALox will determine the inclusion of debug Log Statements by evaluating the (partly standardized) NDEBUG macro that is used by most 'assert' libraries. Hence, if NDEBUG is set, ALOX_DBG_LOG will not be defined.

#define ALOX_DBG_LOG_ON

Explicitly enables debug logging by defining the code selection symbol ALOX_DBG_LOG.

Must not be used in conjunction with ALOX_DBG_LOG_OFF.

If neither ALOX_DBG_LOG_ON nor ALOX_DBG_LOG_OFF are specified, ALox will determine the inclusion of debug Log Statements by evaluating the (partly standardized) NDEBUG macro that is used by most 'assert' libraries. Hence, if NDEBUG is set, ALOX_DBG_LOG will not be defined.

#define ALOX_DLL_EXPORTS

This compiler symbol has to be defined when compiling ALox as a DLL under Windows/MSC. In addition, also ALIB_DLL_EXPORTS has to be defined. Within code units that use the ALox DLL, both symbols must not be defined.

#define ALOX_REL_LOG_CI_OFF

Disables the generation of scope information using the built-in preprocessor macros like "__FILE__" and "__LINE__" for ALox release logging statements

Passing this Symbol to the compiler, makes ALox to leave the code selection symbol ALOX_REL_LOG_CI undefined.
Must not be used in conjunction with ALOX_REL_LOG_CI_ON.

If neither ALOX_REL_LOG_CI_ON nor ALOX_REL_LOG_CIO_OFF are specified, then ALOX_REL_LOG_CI_OFF is the default. In other words: passing the symbol to the compiler is not mandatory.

Note
The ALox Scope Domain mechanism as well as the ALox features Lox.Once and Log Data rely on caller information. These mechanisms are therefore not available to release logging unless ALOX_REL_LOG_CI_ON is set.
#define ALOX_REL_LOG_CI_ON

Explicitly enables the generation of scope information using the built-in preprocessor macros like "__FILE__" and "__LINE__" for ALox release logging statements

Passing this Symbol to the compiler, makes ALox define the code selection symbol ALOX_REL_LOG_CI.
Must not be used in conjunction with ALOX_REL_LOG_CI_OFF.

If neither ALOX_REL_LOG_CI_ON nor ALOX_REL_LOG_CIO_OFF are specified, ALox will disable caller source information for release logging.

Enabling source info for for release logging is seldom wanted. Release executables should not generate log output that an end user is not able to understand. It can make sense however, if release log information from the field goes back to the software development team. Furthermore, it can be very helpful to enable scope information for release Log Statements in the debug version of a compilation unit.

Note
The ALox Scope Domain mechanism as well as the ALox features Lox.Once and Log Data rely on caller information. These mechanisms are therefore not available to debug logging unless this symbol is set.
#define ALOX_REL_LOG_OFF

Explicitly disables release logging by leaving the code selection symbol ALOX_REL_LOG undefined. Any release Log Statements that are embedded in code by using the ALox Macros For Release Logging, will get pruned.

Must not be used in conjunction with ALOX_REL_LOG_ON.

If neither ALOX_REL_LOG_ON nor ALOX_REL_LOG_OFF are specified, release logging implemented by ALox release log macros is enabled. Therefore, this symbol is provided only for completeness. There is no real need to pass it to the compiler.

Remember, that any release logging that is not implemented using the ALox release logging macros, but instead using the ALox classes in the source code directly, is not affected by this macro.

Must not be used in conjunction with ALOX_REL_LOG_OFF.

If neither ALOX_REL_LOG_ON nor ALOX_REL_LOG_OFF are specified, release logging implemented by ALox release log macros is enabled. Therefore, this symbol is provided only for completeness. There is no real need to pass it to the compiler.

Remember, that any release logging that is not implemented using the ALox release logging macros, but instead using the ALox classes in the source code directly, is not affected by this macro.
